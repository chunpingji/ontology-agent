# Feature Specification: 导航信息架构重构 —— 按"本体→实体→应用→治理"分层

**Feature Branch**: `004-navigation-ia-restructure`

**Created**: 2026-06-23

**Status**: Draft

**Input**: User description: "前端信息架构(IA)重构:把左侧导航从 6 项平铺改为按本体(TBox)/实体(ABox)/图谱应用/治理分层;文档抽取并入实体管理做子 Tab;推理控制台并入应用分析;抽出独立审批中心页;复用既有后端,不改推理与合规内核。"

## User Scenarios & Testing *(mandatory)*

平台现有左侧导航为 6 项纯平铺(本体编辑器 · 实体管理 · 文档抽取 · 推理控制台 · 知识图谱 · 事实源),缺少"总览"入口,且 QA 签批能力被埋在推理控制台的弹窗里、无独立治理入口。本特性按领域心智模型(本体 TBox → 实体 ABox → 图谱应用 → 治理)重组导航与页面归属,使每类工作都有明确落点。

### User Story 1 - 分层导航与总览入口 (Priority: P1)

任一角色用户打开平台,左侧看到按领域分层的导航:**总览 / 本体工作台 / 图谱管理(下含 实体管理、应用分析) / 事实源 / 审批中心**;点击"总览"进入平台概览落地页,而不再只能靠左上角 Logo 回首页。

**Why this priority**: 这是整个重构的骨架——其余页面的归属都依赖这个分层结构;即便子页面暂沿用既有内容,分层后的导航本身已显著降低顶层认知负担,可独立交付 IA 价值。

**Independent Test**: 仅实现导航骨架(指向既有/重命名后的路由),即可验证:所有菜单项可点击且解析到对应页面、分组层级正确展开、当前位置高亮正确、"总览"作为一级项可达。

**Acceptance Scenarios**:

1. **Given** 用户在任意页面, **When** 查看左侧导航, **Then** 看到 5 个顶层入口(总览、本体工作台、图谱管理、事实源、审批中心),其中"图谱管理"含两个子项(实体管理、应用分析)。
2. **Given** 用户在导航中, **When** 点击"总览", **Then** 进入平台概览落地页(非仅 Logo 跳转)。
3. **Given** 用户处于某子页面(如实体管理), **When** 观察导航, **Then** 对应顶层分组与子项均呈选中态。
4. **Given** 用户访问平台, **When** 浏览顶层导航, **Then** 不再出现"文档抽取""推理控制台""知识图谱"三个独立顶层平铺项(它们已归位到分组/子页)。

---

### User Story 2 - 文档抽取归入实体管理(一级子 Tab) (Priority: P1)

分析师在"图谱管理 › 实体管理"内,通过一级子标签在"实体浏览/检索"与"文档抽取"之间切换;文档抽取承载完整的"抽取 → 对齐 → 持久化"ABox 充实流水线,作为高频日常主流程保持显眼、不被降级埋深。

**Why this priority**: 文档抽取是日常最高频的 ABox 充实入口,概念上本就属于实体管理(从文档产生候选实体→对齐→入库)。归位后心智一致,且通过子 Tab 保持一级可达性。

**Independent Test**: 进入实体管理,切到"文档抽取"子 Tab,完整执行:新建抽取任务 → 实时查看进度 → 复核候选(合并/拆分)→ 持久化为实体,全程不离开实体管理分区。

**Acceptance Scenarios**:

1. **Given** 用户在实体管理, **When** 查看分区内导航, **Then** 至少有"实体浏览/检索"与"文档抽取"两个一级子标签。
2. **Given** 用户在文档抽取子 Tab, **When** 新建并运行一个抽取任务, **Then** 可实时看到进度、复核候选并最终持久化,功能与重构前等价。
3. **Given** 一个抽取任务进度流正在进行, **When** 用户在实体管理的子 Tab 间切换后返回, **Then** 任务状态可继续查看(不丢失正在进行的工作)。

---

### User Story 3 - 独立审批中心(QA 治理工作台) (Priority: P1)

QA 角色用户点击"审批中心",进入一个聚合治理页面:查看待签结论列表、执行 Part 11 电子签批、执行 QA 拒绝、验真审计哈希链——不再需要先进入推理控制台、再从弹窗里完成签批。

**Why this priority**: 003 已交付完整审批工作流(待签/签批/拒绝/审计链),但前端无独立入口、签批只能在推理页弹窗触发。把治理能力提为一级入口,让 QA 角色拥有明确工作台,显著提升合规操作的可发现性与效率。

**Independent Test**: 以 qa 身份打开"审批中心",看到待签结论列表;对一条结论执行签批或拒绝;切到审计视图执行链验真——以上均在该页内完成,无需进入推理控制台。

**Acceptance Scenarios**:

1. **Given** qa 用户在审批中心, **When** 打开页面, **Then** 看到待签结论列表(含必要的结论上下文)。
2. **Given** qa 用户选中一条待签结论, **When** 执行电子签批(重认证 + 含义/时间/签名人), **Then** 该结论流转为生效态并留痕。
3. **Given** qa 用户选中一条待签结论, **When** 执行 QA 拒绝, **Then** 该结论流转为拒绝终态、其被抑制动作作废并留痕。
4. **Given** qa 用户在审批中心, **When** 触发审计链验真, **Then** 看到链连续性结论(若断裂则定位首个断点)。
5. **Given** 非 qa 角色用户, **When** 尝试进入审批中心或其签批/拒绝操作, **Then** 被拒绝或这些治理操作对其不可见。

---

### User Story 4 - 应用分析整合(推理 + 图谱查询) (Priority: P2)

分析师在"图谱管理 › 应用分析"内,集中使用"知识图谱的应用"类能力:PDE/MACO 计算器、合规评估、规则浏览、结论查看,以及图谱查询/统计(SPARQL、KG 统计)——它们共同表达"对已建图谱的应用",不再散落为两个独立顶层项。

**Why this priority**: 在导航骨架(US1)落地后做整合即可,价值偏向认知聚合而非新增能力;且涉及把现有两页(推理控制台、知识图谱)合并,改动面较大,优先级次于 P1 的归位与治理入口。

**Independent Test**: 打开"应用分析",在同一分区内完成一次 PDE/MACO 计算或合规评估,并执行一次 SPARQL 查询/查看 KG 统计,功能与重构前等价。

**Acceptance Scenarios**:

1. **Given** 用户在应用分析, **When** 查看分区内容, **Then** 同时可达推理能力(PDE/MACO/评估/规则/结论)与图谱查询/统计(SPARQL/stats)。
2. **Given** 用户在应用分析, **When** 执行一次推理计算与一次图谱查询, **Then** 二者均返回与重构前等价的结果。

---

### Edge Cases

- **旧链接/书签**:用户访问重构前的旧路径(如指向"文档抽取""推理控制台""知识图谱"的旧地址),系统应将其导向新位置,不出现硬性 404。
- **角色越权**:非 qa 用户直达审批中心地址时,治理操作不可见/被拒。
- **子 Tab 深链**:直接以带子标签的地址进入(如实体管理的文档抽取子 Tab),应正确落在该子标签。
- **导航高亮**:位于子 Tab 或分组子项时,顶层分组与子项的选中态均应正确。
- **空状态**:无待签结论、无抽取任务、无查询结果时,各页给出明确空状态而非空白。
- **浏览器前进/后退**:在子 Tab 间切换后使用浏览器前进/后退,应回到对应子标签状态。
- **进行中的实时流**:抽取进度流进行中切换分区再返回,工作不丢失。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 左侧导航 MUST 呈现分层结构,顶层为五项:总览、本体工作台、图谱管理、事实源、审批中心;其中"图谱管理"MUST 含两个子项:实体管理、应用分析。
- **FR-002**: "总览"MUST 是一个可点击的一级导航项,进入平台概览落地页(不再仅由 Logo 链接回首页)。
- **FR-003**: "本体工作台"MUST 提供与现"本体编辑器"等价的 TBox 编辑能力(可重命名,能力不减)。
- **FR-004**: "实体管理"MUST 以一级子标签形式同时容纳"实体浏览/检索"与"文档抽取";文档抽取 MUST NOT 被降级为深层隐藏入口。
- **FR-005**: 文档抽取子标签 MUST 保留完整流水线能力:新建任务、实时进度、候选复核(合并/拆分)、持久化为实体——与重构前功能等价。
- **FR-006**: "应用分析"MUST 聚合推理能力(PDE/MACO 计算、合规评估、规则、结论)与图谱查询/统计(SPARQL、KG 统计)。
- **FR-007**: 系统 MUST 提供独立的"审批中心"页面,聚合:待签结论列表、Part 11 电子签批、QA 拒绝、审计链验真。
- **FR-008**: QA 电子签批能力 MUST 可从审批中心直接发起;签批 MUST NOT 仅以推理控制台内嵌弹窗作为唯一入口。
- **FR-009**: 审批中心及其签批/拒绝/验真操作 MUST 受 qa 角色门禁;非 qa 角色 MUST NOT 看到或执行这些治理操作。
- **FR-010**: 重构后所有重构前已有能力 MUST 仍可达(零能力丢失);顶层 MUST NOT 再出现"文档抽取""推理控制台""知识图谱"独立平铺项。
- **FR-011**: 指向重构前旧位置的链接/书签 MUST 被导向其新位置,MUST NOT 产生硬性 404。
- **FR-012**: 导航当前位置 MUST 正确高亮,包含分组、子项与子标签层级。
- **FR-013**: 左侧导航 MUST 在所有页面持续一致呈现(持久侧栏)。
- **FR-014**: 本特性 MUST 复用既有后端接口,不要求新增后端能力;若聚合页需要,仅以既有端点的组合/编排实现。
- **FR-015**: 重构 MUST 不改变后端推理内核、合规哈希链与 Part 11 签名机制的行为(纯前端 IA 与路由层重组)。

### Key Entities *(include if feature involves data)*

- **导航分区 (Nav Group)**:顶层分组(如"图谱管理"),含标题与一组子项;对应一类领域职责。
- **导航项 (Nav Item)**:可点击的导航条目(顶层或子项),映射到一个页面/路由与一个领域能力。
- **分区子标签 (Section Tab)**:页面内的一级标签(如实体管理下的"实体浏览""文档抽取"),在同一分区内切换视图而不改变所属分组。
- **能力归属映射 (Capability Mapping)**:重构前页面/能力 → 重构后落点的对应关系,用于保证零能力丢失与旧链接重定向。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 顶层导航项从 6 个平铺缩减为 5 个分层入口(总览、本体工作台、图谱管理、事实源、审批中心),且"图谱管理"下二级清晰可见。
- **SC-002**: qa 用户从进入平台到在单一专属页面完成一次签批或拒绝,操作路径 ≤ 2 次点击(重构前需进入推理控制台并打开弹窗)。
- **SC-003**: 文档抽取在实体管理内 ≤ 1 次点击可达(子标签),且"抽取 → 复核 → 持久化"全流程可在不离开实体管理分区的情况下完成。
- **SC-004**: 重构前 100% 的既有能力在重构后仍可达;全站 0 个失效链接(死链)。
- **SC-005**: 指向旧路由的既有书签 100% 可解析(重定向到新位置),0 个硬性 404。
- **SC-006**: 在一次轻量可用性检查中,≥ 80% 的用户能按"本体→实体→应用→治理"心智正确定位某项任务应去的菜单位置。

## Assumptions

- **命名定稿**:分组名采用"图谱管理"(替代易与同名可视化页冲突的"知识图谱");推理+查询的子项采用"应用分析"(替代字面易误解为"管理一堆应用"的"应用管理")。该命名已在前期评估中与用户达成一致。
- **后端复用**:既有端点(合规 `signatures/pending`、签批、拒绝、`audit/verify`;推理 评估/计算/规则/结论;抽取 任务/候选;实体;KG 统计/SPARQL;事实源)足以支撑各聚合页,不新增后端能力。
- **旧路由处理**:对旧路径采用重定向到新位置以保护书签(而非直接移除)。
- **签批入口**:审批中心为签批/拒绝的规范主入口;推理/结论视图可保留跳转到审批中心的链接,但不再以内嵌弹窗作为唯一签批途径。
- **样式与身份**:沿用现有 Tailwind 风格,不做视觉品牌改版;角色身份沿用既有可信网关头(`X-User`/`X-Role`)机制,qa 门禁复用既有模式。
- **布局形态**:保留单一持久左侧栏;分组以可折叠/分区标题形式呈现。
- **范围边界**:设计稿/视觉探索(如 palantir-blue 概念图)不在本特性范围,另行处理。

## Out of Scope

- 后端推理内核、合规哈希链、Part 11 签名机制的任何行为变更。
- 本体权威 TTL 写回或 T-Box 结构变更(宪章 II 不触发)。
- 新增后端 API 能力或数据库结构变更。
- 视觉品牌改版、设计系统替换。
- 移动端适配(若当前未支持)。
