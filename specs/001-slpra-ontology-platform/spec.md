# Feature Specification: 临床药物智能辅助生产平台（SLPRA 本体知识平台）

**Feature Branch**: `001-slpra-ontology-platform`

**Created**: 2026-06-20

**Status**: Draft

**Input**: User description: "@docs/临床药物智能辅助生产平台方案.md — 基于 SLPRA 本体的临床药物共线生产风险评估智能辅助平台，交付三大核心能力：①高级分析师维护核心实体与关系；②多源抽取与实体对齐；③实时事实源对齐与 Action 推理。"

## Overview

临床药物共线生产（Shared-Line Production）面临交叉污染风险评估的信息集成挑战：药品风险属性、设备 GMP 属性、污染途径、清洁验证、设施环境、CFDI 8 种特殊情景等知识分散在 SOP、Excel 台账、ERP/MES/LIMS/CTMS 等异构系统中，难以统一推理与实时决策。

SLPRA 领域本体已形式化沉淀该领域知识，但缺少一个让分析师维护知识模型、从异构源抽取知识、并与实时事实源联动推理的工程化平台。本特性构建该平台，交付三大核心能力，最终目标是为高级分析师、生产操作员与质量保证（QA）人员提供可治理、可追溯、近实时的共线生产风险决策支持。

> **现状基线**：平台已具备只读的本体浏览、A-Box 个体管理、可用的规则与数值推理引擎；但能力一的"编辑/维护"为全栈缺口，能力二的抽取流水线已实现但未接线，能力三的事实源连接器仅为桩（Stub）。本规范定义三大能力的**目标态**，并以缺口闭合为交付重点。

## Clarifications

### Session 2026-06-20

- Q: 本特性的交付范围（覆盖整个路线图，还是聚焦一个可交付增量）？ → A: 本特性 = **能力一（T-Box 知识模型维护）缺口闭合**（US1 及其支撑的元数据持久化、审核/发布、TTL 导入导出）；能力二/三及推理相关合规硬化作为**后续独立特性**，不在本特性范围内。
- Q: 多名分析师并发编辑同一实体时如何处理冲突？ → A: **乐观并发**——保存时按版本/更新时间检测冲突，拒绝过期写入并提示重新加载/合并（不采用悲观锁或后写覆盖）。
- Q: 工作台导出 TTL 时对权威基线（~720 公理）的保真要求？ → A: **外科式合并**——导出仅更新工作台建模的公理，对未建模的构造（注释/批注、外部本体对齐公理、SWRL 等）逐字保留，写入前展示 diff（不采用整体重生成或另写独立文件）。
- Q: 发布/版本化的粒度单元？ → A: **批次化发布**——多项暂存的草稿变更一并审核后作为一个带版本的发布，对应一次 TTL 导出 + 一次 Git 提交（不采用逐对象发布或每次保存即发布）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 高级分析师维护核心实体与关系（Priority: P1）

高级分析师登录平台，在统一工作台中维护领域知识模型：定义和修改核心药物实体、生产区域与设备实体、生产过程实体；定义实体之间的关系（含基数与方向）；定义数据属性（如 PDE、OEL）；定义 Action（含前置/后置条件与参数）；为每个实体绑定权威领域本体标识与上层本体类别；编辑实体上的约束/公理；校验模型健康度；提交审核、发布版本，并将变更导出为权威本体文件纳入版本管理。

**Why this priority**: 这是平台的"治理底座"。当前能力仅为只读浏览，无法编辑、校验或发布，是最关键的缺口。没有可维护的知识模型，抽取的目标 Schema 与推理的定义来源都无法演进。它是其他能力的前提，且独立可交付价值——分析师可立即用它治理已有的领域本体。

**Independent Test**: 仅交付本故事即可独立测试：分析师对一个药物类完成"新建 → 设置标签/父类/字段 → 绑定权威本体标识 → 添加约束 → 校验 → 审核发布 → 导出本体文件并查看与基线差异"的全流程，且全程无需工程人员介入。

**Acceptance Scenarios**:

1. **Given** 分析师已登录并选择了临床共线生产领域，**When** 新建一个药物实体类并设置中英文标签、注释、父类、字段定义，**Then** 系统保存该类、在类层次中可见，并标记为草稿待审核状态。
2. **Given** 一个已存在的实体类，**When** 分析师为其定义一个对象关系（指定来源类、目标类、基数 1:1/1:N/N:1/M:N），**Then** 系统保存关系定义并在图谱视图中以带基数的连线呈现。
3. **Given** 一个实体类，**When** 分析师为其绑定权威领域本体标识与上层本体类别并保存，**Then** 系统记录映射并在映射健康度校验中将该类标记为"已映射"。
4. **Given** 一个实体类上已取到的约束/公理，**When** 分析师查看该类详情，**Then** 约束被完整渲染（而非丢弃），且可进行新增与删除。
5. **Given** 分析师完成一批草稿变更，**When** 执行模型校验，**Then** 系统报告孤立类、未映射字段、与权威基线的漂移以及一致性冲突，并阻止存在阻断性问题的发布。
6. **Given** 校验通过的变更，**When** 分析师提交审核并发布，**Then** 系统将状态推进为已发布、归档版本、导出本体文件并生成与基线的差异，且全过程留痕于审计日志。

### User Story 2 - 共线生产风险评估推理（Priority: P2）

> **本特性范围外 — 后续独立特性**（保留于此作为平台全景与上下文；不在本特性的计划/任务范围内）。

操作员或分析师针对一组共线生产事实（某设备/产线在某时段排产了哪些产品、产品的风险属性、设备材质与可清洁性等）发起风险评估。系统结合知识模型与规则，自动完成药品风险分类、设备专用化决策、污染途径风险评分、CFDI 共线情景识别，计算 MACO/PDE 残留限度，在多规则冲突时以"安全优先"裁决，并输出带完整规则链与法规依据的可追溯结论。

**Why this priority**: 这是平台最终交付给业务的核心价值——共线交叉污染风险决策。它将本体与规则转化为可执行的合规结论，是平台存在的理由。可在能力一的基础上独立验证，且推理引擎已具雏形，闭合后即可交付看得见的业务价值。

**Independent Test**: 给定一组共线事实（例如细胞毒药物与商业药物在同一离心机同时段排产），系统在无人工干预下产出风险分类、专用化决策（含冲突裁决）、污染途径评分、CFDI 情景标识与 MACO/PDE 数值，并可展开每条结论的规则链与法规条款。

**Acceptance Scenarios**:

1. **Given** 一个具有基因毒性/高活性/致敏/β-内酰胺属性的药品，**When** 发起风险评估，**Then** 系统按对应规则自动完成药品风险分类。
2. **Given** 一条触发"无条件专用"（如青霉素）与"可灭活共线"两条相互冲突规则的事实组合，**When** 系统裁决，**Then** 以安全优先原则裁定为"需专用化"，并在结论中标注冲突与裁决依据。
3. **Given** 残留风险、低 PDE 与差可清洁性同时成立，**When** 评估污染途径，**Then** 系统判定为高风险并给出评分。
4. **Given** 一组共线排产组合，**When** 进行情景识别，**Then** 系统正确识别 8 种 CFDI 共线情景中适用的项。
5. **Given** PDE 数据可用，**When** 计算残留限度，**Then** 系统对 PDE 法、千分之一剂量法、10ppm 通用限、NOEL 法取最小值作为 MACO，并优先采用 PDE 法。
6. **Given** 任一已产出的合规结论，**When** 用户请求溯源，**Then** 系统展示触发的规则 ID 链及其对应法规依据。

### User Story 3 - 多源抽取与实体对齐（Priority: P3）

> **本特性范围外 — 后续独立特性**（保留于此作为平台全景与上下文；不在本特性的计划/任务范围内）。

分析师上传设备台账 Excel、工艺规程/SOP Word 文档，或对接数据库，系统自动解析并抽取实体、属性、关系与 Action（含"若…则…必须…"逻辑转化为前置/后置条件），将抽取目标对齐到领域本体类体系，并对跨源描述同一对象（如同一设备在 Excel/MES/ERP 中的不同标识）的实例进行对齐归组。低置信结果进入审核队列，分析师确认、拒绝、合并或拆分后方可发布。

**Why this priority**: 抽取与对齐解决"知识录入"的规模化问题，让平台从已有的异构资料快速冷启动与持续补充。它依赖能力一定义的目标 Schema，价值高但非业务结论的直接来源，故置于 P3。读取器与流水线已实现，主要缺口是接线与审核 UI。

**Independent Test**: 上传一份设备台账 Excel 与一份 SOP Word，系统抽出对应的实体/实例/Action 并对齐到本体类，跨源同一设备被正确归组并标记规范实例，低置信项进入审核队列且分析师可执行确认/合并/拆分。

**Acceptance Scenarios**:

1. **Given** 一份设备台账 Excel，**When** 创建抽取作业，**Then** 系统实际触发抽取流水线（解析→抽取→对齐→审核队列），并通过进度反馈实时展示各阶段进展。
2. **Given** 一份含表格与正文的 SOP Word，**When** 抽取完成，**Then** 表格行被抽为实例、正文中的"若…则…必须…"逻辑被抽为带前置/后置条件的 Action 候选。
3. **Given** 同一设备在多源中以不同标识出现，**When** 执行对齐，**Then** 系统以设备唯一编号为强主键将其归为一组并标记规范实例。
4. **Given** 抽取产生的低置信候选，**When** 分析师在审核界面操作，**Then** 可对候选执行确认、拒绝、手动合并或拆分，确认后方进入知识库。
5. **Given** 抽取目标类体系，**When** 抽取运行，**Then** 抽取提示注入领域受控词表（OEB/PDE/材质/洁净级别），以降低自由文本错误。

### User Story 4 - 实时事实源对齐与 Action 编排（Priority: P3）

> **本特性范围外 — 后续独立特性**（保留于此作为平台全景与上下文；不在本特性的计划/任务范围内）。

平台连接生产排期（APS）、ERP、MES、LIMS、CTMS 等事实源，增量拉取事实（如某设备时段内生产哪些产品、批次、清洁记录、残留检测结果、临床试验用药批次），归一到本体标识并物化为事实实例，发布"事实变更事件"触发增量推理。当推理得出特定结论时，自动编排相应动作：生成专用化工单与告警、创建灭活验证或清洁再验证任务、阻断不相容排期并向排期系统回写建议、生成风险评估报告。决策结果在实时看板上以共线相容性热力图、当前排期风险与推理链溯源呈现。

**Why this priority**: 实时联动与动作编排是平台的"闭环"与最高级形态，价值最大但依赖前序能力齐备且需对接真实外部系统，工程量与外部依赖最高，故置于 P3 末位、按路线图分阶段交付。

**Independent Test**: 配置一个事实源连接器，注入一次事实变更（设备排产组合变化），系统在限定时延内增量重算受影响子图、刷新看板、触发对应动作（如生成专用化工单/阻断排期建议），并可在看板展开推理链。

**Acceptance Scenarios**:

1. **Given** 已配置的事实源连接器，**When** 事实源产生增量变更，**Then** 系统将增量归一到本体标识、物化为事实实例并发布事实变更事件。
2. **Given** 一次事实变更事件，**When** 触发推理，**Then** 系统仅对受影响子图（相关设备/产品/区域）重算，并在不超过 5 秒内刷新结论。
3. **Given** 推理得出"需专用化"结论，**When** 动作编排执行，**Then** 系统生成专用化工单与告警，并记录动作执行留痕。
4. **Given** 排期使不相容产品同设备同时段，**When** 检测到冲突，**Then** 系统标记冲突、阻断该排期并向排期系统回写调整建议。
5. **Given** 一次完成的评估，**When** 用户查看实时看板，**Then** 看板展示设备×产品的共线相容性热力图、未来若干天排期风险，并可展开任一结论的推理链溯源。
6. **Given** 高风险/合规阻断类结论，**When** 生成风险评估报告，**Then** 报告需经 QA 强制复核后方可生效。

### User Story 5 - 合规、审计与权限治理（Priority: P3）

> **本特性部分相关**：仅与**知识模型维护**直接相关的部分（维护操作的审计留痕、对维护/发布操作的角色门禁）纳入本特性范围；推理结论相关的 QA 强制电子签名、ALCOA+ 哈希链等随能力三作为后续特性交付。

平台对全链路（抽取/对齐/推理/动作）操作留痕，审计记录不可篡改并具完整性保护；按角色分级授权——高级分析师可维护知识模型与发布、操作员可查看推理、QA 复核高风险结论；高风险、专用化、合规阻断类结论须 QA 电子签名后方可生效；知识模型版本可追溯并保留至法规条款的映射依据。

**Why this priority**: 制药 GMP 场景下，数据完整性（ALCOA+）、权限分级与强制复核是合规上线的硬性门槛。它横切所有能力，但可在核心能力稳定后作为"合规硬化"阶段集中交付。

**Independent Test**: 以不同角色登录验证权限边界；对一条高风险结论验证"未经 QA 签名不可生效"；对任一关键操作验证审计记录存在且不可篡改；对一次模型发布验证版本归档与法规映射可追溯。

**Acceptance Scenarios**:

1. **Given** 三类角色（分析师/操作员/QA），**When** 各自登录操作，**Then** 系统按角色限制可执行的操作范围。
2. **Given** 一条高风险或合规阻断结论，**When** QA 未签名，**Then** 结论不生效；QA 电子签名后方生效。
3. **Given** 任一抽取/对齐/推理/动作操作，**When** 操作发生，**Then** 系统写入只追加且具完整性保护的审计记录。
4. **Given** 一次知识模型发布，**When** 事后追溯，**Then** 可查到版本归档与对应法规条款映射。

### Edge Cases

- **规则冲突**：多条规则对同一对象给出相反的专用化结论时，系统必须以安全优先（"需专用化"优先）裁决，并保留冲突双方与裁决依据可追溯。
- **缺失关键数据**：当 PDE 等关键数据缺失时，MACO 计算应回退到可用方法（剂量法/通用限/NOEL 法）并明确标注所用方法与数据缺口。
- **抽取无外部模型能力**：当用于逻辑抽取的智能能力不可用时，系统应回退到结构化原始数据抽取而非失败。
- **事实源不可达/超时**：连接器无法连接或超时时，系统应保留上一已知良好状态、记录失败并告警，而非污染知识库。
- **本体不一致**：发布前一致性校验发现矛盾公理（如污染途径互斥被违反）时，应阻止发布并定位冲突。
- **导出漂移**：工作台变更与权威本体文件出现漂移时，导出/差异功能须高亮差异并以外科式合并保留未建模公理，避免静默覆盖或丢弃既有公理（注释/对齐/SWRL 等）。
- **跨源对齐歧义**：同一编号在不同源指向疑似不同实体时，应进入人工审核而非自动合并。
- **并发编辑**：两名分析师同时编辑同一实体时，系统以乐观并发控制避免相互覆盖——保存时按版本/更新时间检测冲突，若检测到过期写入则拒绝该保存并提示分析师重新加载后再合并提交（不静默覆盖）。

## Requirements *(mandatory)*

### Functional Requirements

#### 能力一：知识模型维护（T-Box）

- **FR-001**: 系统 MUST 允许高级分析师对核心药物实体、生产区域与设备实体、生产过程实体进行新建、修改与删除（含中英文标签、注释、父类、字段定义）。
- **FR-002**: 系统 MUST 允许定义、修改与删除实体间关系，含来源类、目标类、基数（1:1/1:N/N:1/M:N）、方向与逆关系。
- **FR-003**: 系统 MUST 允许定义、修改与删除数据属性并指定其数据类型（如每日允许暴露量、职业暴露限值等）。
- **FR-004**: 系统 MUST 允许定义、修改与删除 Action，含执行主体、目标、前置条件、后置条件与参数。
- **FR-005**: 系统 MUST 允许在实体上新增、查看与删除约束/公理（如存在量化、全称量化、基数、互斥、等价），且已存在的约束 MUST 被完整渲染。
- **FR-006**: 系统 MUST 允许为实体类与关系绑定权威领域本体标识与上层本体类别，并支持字段级映射。
- **FR-007**: 系统 MUST 提供模型健康度校验，识别孤立类、未映射字段、与权威基线的漂移以及本体一致性冲突。
- **FR-008**: 系统 MUST 提供审核与发布工作流（草稿 → 审核 → 已发布），含置信度、已审核、停用标记与版本归档。
- **FR-008a**: 发布 MUST 以**批次化发布**为单元——允许暂存多项草稿变更，经审核后作为一个带版本号的发布整体生效，对应一次 TTL 导出与一次 Git 提交；版本归档以发布批次（而非单个对象或单次保存）为粒度。
- **FR-009**: 系统 MUST 支持从权威本体文件导入知识模型、将工作台变更导出为权威本体文件，并展示二者差异（diff）。
- **FR-009a**: 导出 MUST 采用外科式合并，仅更新工作台建模的公理；对工作台未建模的本体构造（注释/批注、外部本体对齐公理、SWRL 规则等）MUST 逐字保留，且 MUST 在写入前向分析师展示 diff 供确认，不得整体重生成或静默丢弃未建模内容。
- **FR-010**: 系统 MUST 提供受控词表/风险属性录入向导（如 OEB、PDE、致敏等级、灭活方式），降低自由文本错误。
- **FR-011**: 系统 MUST 提供知识模型的图谱可视化（实体、Action、带基数的关系）。
- **FR-011a**: 系统 MUST 以乐观并发控制管理对同一知识模型对象的并发编辑——保存时校验版本/更新时间，对过期写入予以拒绝并提示重新加载，不得静默覆盖他人变更。

#### 能力二：多源抽取与对齐（本特性范围外 — 后续独立特性）

- **FR-012**: 系统 MUST 支持从 Excel（设备台账/物料/批记录）抽取实体、属性与实例。
- **FR-013**: 系统 MUST 支持从 Word（SOP/工艺规程）抽取实体/实例（表格）与过程实体/Action（正文逻辑）。
- **FR-014**: 系统 MUST 将"若…则…必须…"类逻辑转化为带前置/后置条件的 Action 候选。
- **FR-015**: 系统 MUST 在创建抽取作业时实际触发抽取流水线（解析→抽取→对齐→审核队列），并提供作业进度反馈。
- **FR-016**: 系统 MUST 将抽取目标对齐到领域本体类体系，并在抽取过程中注入领域受控词表以降低错误。
- **FR-017**: 系统 MUST 对跨源描述同一对象的实例进行对齐归组并标记规范实例（设备以唯一编号为强主键，药品以活性成分+剂型+规格为对齐键）。
- **FR-018**: 系统 MUST 将低置信抽取/对齐结果送入人工审核队列，分析师可确认、拒绝、合并或拆分，确认后方进入知识库。
- **FR-019**: 系统 SHOULD 支持从数据库抽取（表结构→实体、外键→关系、状态/时间戳→Action）。

#### 能力三：实时事实源对齐与推理（本特性范围外 — 后续独立特性）

- **FR-020**: 系统 MUST 提供事实源连接能力，支持生产排期、ERP、MES、LIMS、CTMS 等来源的事实接入。
- **FR-021**: 系统 MUST 将事实源增量归一到本体标识、物化为事实实例，并发布事实变更事件。
- **FR-022**: 系统 MUST 在事实变更后仅对受影响子图进行增量重算。
- **FR-023**: 系统 MUST 自动完成药品风险分类（基因毒/高活性/致敏/β-内酰胺等）。
- **FR-024**: 系统 MUST 自动完成设备专用化决策（青霉素无条件专用、细胞毒+不可灭活专用、高暴露等级专用、激素独立暖通等）。
- **FR-025**: 系统 MUST 自动完成污染途径风险评分。
- **FR-026**: 系统 MUST 自动识别 8 种 CFDI 共线情景中适用的项。
- **FR-027**: 系统 MUST 在多规则冲突时以安全优先（"需专用化"优先）裁决。
- **FR-028**: 系统 MUST 计算残留限度，对 PDE 法、千分之一剂量法、10ppm 通用限、NOEL 法取最小值，并在 PDE 可用时优先采用 PDE 法。
- **FR-029**: 系统 MUST 为每条结论保存触发规则 ID 链及对应法规依据，并支持展开溯源。
- **FR-030**: 系统 MUST 在事实变更后达成相应动作编排：生成专用化工单/告警、创建灭活验证或清洁再验证任务、阻断不相容排期并回写建议、生成风险评估报告。
- **FR-031**: 系统 MUST 提供实时推理看板，含共线相容性热力图、当前排期风险与推理链溯源。

#### 跨切：合规、审计与权限

> 本组中与**知识模型维护**直接相关的 FR-032（维护/发布留痕）、FR-033（维护/发布角色门禁）、FR-035（模型版本可追溯）纳入本特性范围；FR-034（推理结论 QA 强制电子签名）随能力三为后续特性。

- **FR-032**: 系统 MUST 对知识模型维护与发布的全链路操作留痕，审计记录只追加且具完整性保护。
- **FR-033**: 系统 MUST 提供基于角色的访问控制，至少区分高级分析师、操作员、QA 三类权限；其中知识模型的编辑与发布操作 MUST 限定为高级分析师角色。
- **FR-034**: 系统 MUST 对高风险/专用化/合规阻断类结论强制 QA 电子签名后方可生效。*（本特性范围外 — 随能力三交付）*
- **FR-035**: 系统 MUST 保证知识模型版本可追溯，并保留至法规条款的映射依据。

### Key Entities *(include if feature involves data)*

- **实体类（Class）**：知识模型中的概念类型（药物、设备、生产区域、生产过程等），含标签、注释、父类、字段定义、置信度、审核/停用状态、权威本体标识与上层本体类别映射。
- **关系（LinkType / 对象属性）**：连接两个实体类的关系定义，含来源类、目标类、基数、方向、逆关系。
- **数据属性（Data Property）**：实体类上的数据型属性及其数据类型（如每日允许暴露量、职业暴露限值、可清洁性评分）。
- **Action（动作定义）**：含执行主体、目标、前置条件、后置条件与参数；以及由推理触发的具体动作（专用化工单、灭活/清洁再验证任务、排期阻断、报告生成）。
- **约束/公理**：实体上的逻辑约束（存在/全称量化、基数、互斥、等价）。
- **实体实例（A-Box 事实）**：知识库中的具体个体（某台设备、某个药品批次），由抽取或事实源物化产生。
- **抽取作业与候选**：抽取配置、作业、低置信候选及其审核状态。
- **风险评估结论**：含分类结果、专用化决策、污染途径评分、CFDI 情景、MACO/PDE 数值、规则链与法规依据、冲突裁决记录。
- **事实源连接器配置**：外部系统（排期/ERP/MES/LIMS/CTMS）的接入配置与同步水位。
- **审计记录**：全链路操作留痕，只追加并具完整性保护。
- **用户与角色**：高级分析师、操作员、QA 及其权限与电子签名。
- **CFDI 共线情景**：8 种受监管的共线生产特殊情景。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 高级分析师可在无工程人员介入下，独立完成对实体类/关系/数据属性/Action/约束的增改删并发布，且发布变更可导出为权威本体文件并查看与基线差异。
- **SC-002**: 关键实体类 100% 绑定权威领域本体标识与上层本体类别，模型健康度校验无孤立项。
- **SC-003**: 设备台账 Excel 与 SOP Word 抽取后，关键字段抽取准确率达到约定阈值（建议 ≥ 90%），且作业进度对用户可见。*（范围外 — 能力二，后续独立特性）*
- **SC-004**: 跨源描述同一设备/药品的实例被正确归组，规范实例标记正确，且分析师可对候选执行合并/拆分。*（范围外 — 能力二，后续独立特性）*
- **SC-005**: 8 种 CFDI 共线情景被自动正确识别；青霉素/细胞毒/高暴露等级专用化决策正确；规则冲突一律以安全优先裁决。*（范围外 — 能力三，后续独立特性）*
- **SC-006**: MACO 计算对四种方法正确取最小值，且 PDE 可用时优先采用 PDE 法。*（范围外 — 能力三，后续独立特性）*
- **SC-007**: 事实源发生变更后，相关风险结论刷新时延 ≤ 5 秒（近实时）。*（范围外 — 能力三，后续独立特性）*
- **SC-008**: 每条合规结论 100% 可展开其规则链与对应法规依据（决策可追溯）。*（范围外 — 能力三，后续独立特性）*
- **SC-009**: 全链路知识模型维护/发布关键操作 100% 有不可篡改审计记录（**本特性范围内**，对应 FR-032/035）；高风险/合规阻断类**推理结论**在 QA 电子签名前 100% 不生效部分*（范围外 — 随能力三，对应 FR-034）*。
- **SC-010**: 角色权限边界生效——非授权角色无法执行模型维护/发布等受限操作。

## Assumptions

- **范围**：本特性**仅交付能力一（知识模型 / T-Box 维护）的缺口闭合**——US1 及其支撑能力（可编辑元数据持久化、SLPRA IRI/BFO 映射、约束编辑、模型健康度校验、审核/发布工作流、TTL 导入/导出/diff、维护操作的审计留痕与角色门禁）。**范围外（后续独立特性）**：能力二（多源抽取与对齐，US3 / FR-012–FR-019）、能力三（实时事实源对齐与 Action 推理，US2 与 US4 / FR-020–FR-031）、以及推理结论相关的合规硬化（QA 强制电子签名 FR-034 等）。方案文档中的全平台愿景在本规范中保留为上下文，但不纳入本特性的计划与任务。
- **知识来源权威性**：领域本体文件是知识模型的"真理来源"，平台与之双向映射并通过版本管理保持一致。
- **人在环**：抽取/对齐结果均带置信度，须经分析师审核方可发布；高风险合规结论强制 QA 复核。
- **冲突解决策略**：多规则冲突时以"需专用化"安全优先，与法规"合规优先"一致。
- **近实时时延目标**：事实变更到结论刷新的目标时延为 ≤ 5 秒。
- **抽取准确率阈值**：在未另行约定时，关键字段抽取准确率目标取业界常见基线（建议 ≥ 90%），最终阈值由验收时与业务方确认。
- **合规框架**：以制药 GMP 的数据完整性（ALCOA+）、CFDI 共线生产监管要求为合规基线。
- **既有基线**：平台已具备只读本体浏览、A-Box 个体管理与可用的规则/数值推理引擎；本规范以闭合编辑/接线/实时联动等缺口为交付主线。
- **用户群体**：主要用户为制药企业的高级分析师、生产操作员与 QA 人员，具备相应领域知识。
