# Feature Specification: 前端组件系统重构（Tailwind + shadcn/ui）

**Feature Branch**: `005-shadcn-ui-refactor`

**Created**: 2026-06-23

**Status**: Draft

**Input**: User description: "补充：使用tailwind css库和shadcn组件库重构"

## 背景与约束 *(Context & Constraints)*

本特性是 **004（导航 IA 重构）** 的自然续作。004 在非目标中明确写道「不做视觉品牌改版（沿用现有 Tailwind 风格，设计稿探索另行处理）」——本特性正是承接那块被推迟的工作：在**不改变信息架构、路由与后端契约**的前提下，把当前**逐屏手写、硬编码**的样式收敛为**一套统一、可复用、可访问的组件系统**。

> **关于「不得渗入实现细节」**：本特性的对象本身就是「采用 shadcn/ui 组件库 + Tailwind CSS」这一**组件体系选择**，因此 shadcn/ui、Tailwind、设计令牌等名词是本规范的**主题**而非泄漏的实现细节。功能需求（FR）仍以**用户/维护者可观察的结果**（一致性、可访问性、可主题化、可复用、行为零回归）表述；具体库名仅出现在本节、假设与依赖中。

**纯前端特性**：不新增后端端点、不改数据模型、不改 API 调用与载荷。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 设计系统基座 + 应用外壳统一 (Priority: P1)

平台引入**一套集中的设计令牌（颜色/圆角/间距/排版/层级）与基础组件原语**，并率先应用到**始终在场的应用外壳**：左侧领域分层导航、当前激活高亮、身份切换器、Logo、各分区页面的标题/容器骨架。任何用户进入任意路由，都会立即继承到统一、专业、可访问的界面框架。

**Why this priority**: 外壳在每一屏都出现；先统一它能让全站立刻获得一致观感，并且令牌 + 原语是后续所有迁移的**前置依赖**。仅完成本故事即构成一个可演示、有价值的 MVP。

**Independent Test**: 加载任意路由，外壳（侧栏、导航高亮、身份切换、Logo）均由统一组件系统渲染，视觉一致；键盘可遍历、读屏可识别；004 的导航行为（高亮规则、分组展开、身份切换、旧链接重定向）**零回归**。

**Acceptance Scenarios**:

1. **Given** 用户位于任一受保护路由, **When** 页面渲染, **Then** 侧栏、导航项、身份切换器、页面容器全部来自统一组件系统，且颜色/间距/排版来自集中令牌而非内联硬编码值。
2. **Given** 用户使用键盘操作侧栏与身份切换器, **When** 用 Tab / 方向键 / Enter 操作, **Then** 焦点可见、可达、可选择，读屏可朗读控件名称与状态。
3. **Given** 用户访问 004 旧平铺路由（如 `/extraction`）, **When** 触发 308 重定向, **Then** 导航与高亮行为与 004 完全一致，未因重构而改变。

---

### User Story 2 - 高频交互控件统一（按钮 / 卡片 / 表格 / 标签页 / 表单 / 弹窗） (Priority: P2)

把 004 各高流量界面上反复出现的交互原语迁移到统一组件：总览的概览卡 + 快捷入口卡、实体页子标签、应用分析的页内标签、审批中心的待签/审计表格、所有按钮、QA 电子签批弹窗与 QA 拒绝弹窗、表单输入与下拉选择。

**Why this priority**: 这些是用户接触最频繁的交互；统一后它们立刻获得一致外观与**真正的可访问性**（弹窗焦点陷阱、Esc 关闭、可访问的标签页与下拉）。依赖 US1 的基座，但可独立验证与交付。

**Independent Test**: 走查 总览 / 实体 / 应用分析 / 审批中心 —— 卡片、标签页、表格、按钮、弹窗均使用统一组件；弹窗能陷阱焦点并支持 Esc 关闭、关闭后焦点归还触发元素；标签页可键盘切换；**所有原有行为保持一致**（含 Part 11 重认证与校验、审计链验真、抽取流断点续传）。

**Acceptance Scenarios**:

1. **Given** QA 用户在审批中心点击「签批」, **When** 电子签批弹窗打开, **Then** 焦点被陷阱在弹窗内、Esc 可关闭、关闭后焦点回到「签批」按钮，且 Part 11 重认证与原因校验行为不变。
2. **Given** 用户在应用分析切换「推理 / 图谱查询」标签, **When** 用鼠标或键盘切换, **Then** 标签为可访问的标签页控件，激活态由令牌驱动，内容切换行为与 004 一致。
3. **Given** 总览/审批中心存在列表数据, **When** 列表为空或有数据, **Then** 空态与表格均由统一组件呈现，且数据获取与角色门控逻辑未变。

---

### User Story 3 - 领域重屏（本体 / 治理 / 事实源 / 抽取）控件收敛 (Priority: P3)

把剩余的领域密集型界面迁移到统一组件系统，淘汰最后一批临时/硬编码样式：本体工作台各面板（类层次、关系、数据属性、约束编辑、风险向导、TTL 工具条、冲突弹窗等）、事实源对齐与实时推理面板、文档抽取的创建表单 / 进度 / 对齐复核。

**Why this priority**: 这些屏交互复杂、迭代较少，是硬编码样式残留最多之处；最后收敛可在不阻塞高频路径的前提下达成**全站令牌 + 组件覆盖**。

**Independent Test**: 本体 / 事实源 / 实体-抽取 各屏完全由统一组件与令牌渲染；被迁移控件不再残留硬编码颜色工具类；各面板的编辑/保存/校验/diff 预览等行为与重构前一致。

**Acceptance Scenarios**:

1. **Given** 用户在本体工作台编辑某面板, **When** 触发保存/校验/冲突, **Then** 表单、按钮、对话框均为统一组件，且 TTL 外科式合并、diff 预览等领域行为不变。
2. **Given** 用户打开事实源连接器管理或实时推理面板, **When** 渲染, **Then** 控件全部来自统一组件系统，无遗留临时样式。

---

### Edge Cases

- **角色门控视图**：审批中心仅 QA 可见——统一组件须在 `senior_analyst` / `operator` / `qa` 各角色下都正确渲染（含非 QA 的占位提示态）。
- **合规弹窗零回归**：签批 / 拒绝弹窗迁移后须**完整保留** 21 CFR Part 11 重认证、原因必填校验与提交流程，不得弱化。
- **SSR / 水合**：主题与组件须以 CSS 变量驱动，避免服务端/客户端渲染不一致（hydration mismatch）。
- **d3 图谱可视化**：`graph-visualization` 的 SVG/canvas 内部渲染**不在组件库范畴**，仅迁移其外围 chrome（容器、工具条、按钮）。
- **键盘/焦点不回退**：引入可访问组件不得破坏既有键盘流；既有可聚焦元素的 Tab 顺序须保持合理。
- **空态与加载态**：列表/统计在数据为空或加载中时，统一组件须给出明确空态/骨架，而非裸文本或布局抖动。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 前端 MUST 在所有现有界面（应用外壳、本体 TBox、实体 ABox、应用分析、治理审批、事实源）呈现**单一、一致的视觉组件系统**，取代当前逐屏各异的临时样式。
- **FR-002**: 反复出现的 UI 控件（按钮、卡片、表格、标签页、对话框、表单输入、下拉选择、徽标/标签、提示/空态）MUST 由**共享、可复用的组件**提供，而非在各处复制内联标记。
- **FR-003**: 交互式浮层组件（对话框/模态、下拉/选择菜单、标签页）MUST 可访问——可键盘操作、模态须做焦点管理（焦点陷阱 + 关闭后焦点归还）、并具备读屏可识别的角色与标签。
- **FR-004**: 视觉样式（颜色、表面、圆角、间距、排版层级）MUST 由**集中的设计令牌**驱动，而非散落在各组件中的硬编码值；单点修改令牌须能传播到全站。
- **FR-005**: 重构 MUST 完整保留 004 的信息架构、路由、重定向与导航行为，**不改变任何路由或 IA**。
- **FR-006**: 重构 MUST 保留全部既有功能行为与后端契约——数据获取、角色门控、Part 11 签批/拒绝流、审计验真、抽取流式——**不改变任何 API 调用或载荷**。
- **FR-007**: 角色门控与条件 UI（如 QA-only 审批中心、身份切换器）MUST 在迁移后于各角色下正确渲染。
- **FR-008**: 应用 MUST 能成功构建并通过 lint，且**不引入相对重构前基线的新错误**（与 004 同等质量门禁）。
- **FR-009**: 迁移后的界面 MUST 不产生服务端/客户端水合不一致。
- **FR-010**: 视觉结果 MUST 与当前观感保持**基本一致**（简洁、中性、专业）——本特性是**组件体系迁移**，**非品牌视觉改版**（品牌改版仍属非目标，沿 004）。
- **FR-011**: 组件系统 MUST 支持**增量迁移**——US1/US2/US3 可分别独立交付，且任一时刻新旧控件可短期共存而不破坏页面。

### Key Entities *(设计系统概念制品)*

- **设计令牌 (Design Token)**：集中定义的视觉变量（颜色/表面/圆角/间距/排版/层级），全站唯一来源，决定主题外观。
- **共享 UI 组件 (Shared UI Component)**：从令牌派生、封装可访问行为的可复用控件（按钮、卡片、表格、标签页、对话框、表单控件等）。
- **主题 (Theme)**：一组令牌取值的集合（v1 为单一浅色主题）；结构上预留多主题/暗色扩展位但本次不实现。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 完成全部故事后，**100% 现有界面（9 条路由）与高频控件**经由统一组件系统渲染——零界面停留在旧的逐屏临时样式上。
- **SC-002**: 全部交互浮层（对话框、下拉/选择、标签页）**完全可键盘操作**并通过基础可访问性检查（焦点陷阱、Esc 关闭、ARIA 角色）——被迁移交互控件上**严重可访问性问题数 = 0**。
- **SC-003**: 视觉样式集中化——组件标记中**散落的硬编码颜色值收敛至约 0**（全部由令牌驱动）；单点更改主题令牌即可全站生效。
- **SC-004**: **功能与契约零回归**——004 的全部用户旅程（导航、总览统计、实体浏览/抽取、分析标签、QA 签批/拒绝、审计验真）迁移后行为不变、全部通过。
- **SC-005**: 构建 + lint 门禁相对重构前基线**不引入新错误**。
- **SC-006**: 新增任一共享控件可在 **≥2 个界面**直接复用而无需重写样式——验证组件系统的复用价值。

## Assumptions

- 本特性是在**既有 Tailwind CSS** 之上**采用 shadcn/ui 组件库**的「组件体系 + 视觉一致性」重构；所列库名是本特性的**显式主题**，非泄漏实现细节。
- 目标为**视觉基本一致 + 轻度打磨**；**完整品牌/视觉改版仍属非目标**（沿 004 推迟，另行处理）。
- 迁移覆盖**当前整个前端面**（9 条路由 + 20 个组件中的高频控件），按优先级故事**增量交付**；仅完成 US1 即构成可用 MVP。
- **暗色模式 / 多主题不在本次范畴**；但令牌体系按可后续无重写地扩展暗色来组织。
- 004 的 IA、路由、重定向与后端 API 契约是**稳定输入**，原样复用。
- d3 图谱可视化内部渲染不在组件库范畴，仅迁移外围 chrome。
- **无需新增后端端点或数据模型变更**（纯前端特性）。

## Dependencies

- 依赖 **004（导航 IA 重构）** 已落地：本特性重构的正是 004 交付的外壳、页面与组件。
- 复用既有第三方能力作为输入（既有 Tailwind 配置、`clsx` / `tailwind-merge` / `lucide-react` 等已在用依赖）。
