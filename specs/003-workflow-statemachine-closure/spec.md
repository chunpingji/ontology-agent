# Feature Specification: 分析结论工作流状态机闭环 —— 结论流水线自举与显式生命周期

**Feature Branch**: `003-workflow-statemachine-closure`

**Created**: 2026-06-22

**Status**: Draft

**Input**: User description: "@docs/workflow-statemachine-design.md"

## Overview

`002` 这一轮以**结论中心（conclusion-centric）的合规/动作流水线**落地了能力三的合规硬化——审计哈希链 ✅、QA 电子签名 ✅、Action 编排引擎 ✅、增量重算 + 取代链 ✅、风险评估报告导出 ✅。但 `docs/workflow-statemachine-design.md`（v0.2 as-built 对照）核出该流水线**尚不能端到端自举**，存在三处接线缺口，使"分析思路工作流"无法在无人工补数据的情况下从头跑通：

- **G1 —— 结论无初始落库路径**：无状态评估（"评估"动作）只返回结果、不持久化；唯一实例化结论的入口在增量重算中，且只能从**已存在**的结论刷新而来。没有"第一条结论"，下游的复核、QA 签批、动作派发、报告导出**全部无源头**。
- **G2 —— 强制 QA 闸门未自动 arm**：除模型默认与增量沿用旧值外，**无任何据风险将结论置入"待签"态的判定**。高风险结论不会自动进入待签，Part 11 合规闸门形同虚设。
- **G3 —— 事实事件 → 增量重算未自动订阅**：物化层已发布携子图的事实变更事件，但**无订阅者**桥接到重算；近实时重算只能手动触发，物化与重算之间断链。

此外，结论的"状态"目前是**隐式**的，分散在三处标志上，迁移逻辑硬编码、各处 `if` 分支自守，缺乏统一的状态枚举与集中守卫，非法跃迁不被一致拒绝。

本特性采纳设计文档 §6 的**路线 A（务实闭环）**：补齐 G1/G2/G3 让现有结论中心流水线端到端跑通，并把隐式状态固化为**显式生命周期状态 + 集中式迁移合法性校验**。本特性**不**追求路线 B（把迁移定义数据化为本体库可治理的 `OntologyAction`）——该项作为后续特性，待路线 A 跑通、需求稳定后再评估。

> **现状基线（经设计文档逐条核验）**：`ReasoningExecution`/`ActionExecution`/`ElectronicSignature` 模型、全局哈希链审计、Action 编排、增量重算与取代链、QA 签名 API、报告导出均已存在；缺的是把它们串成可自举闭环的三根线（G1/G2/G3）与一层显式状态治理。本规范以"接通源头 + 显式化状态机"为交付主线，不重写推理规则与合规机制。

## Clarifications

### Session 2026-06-22

- Q: 结论显式生命周期状态的枚举全集？ → A: **四态** —— `待签(pending_signature)` / `生效(effective)` / `已取代(superseded)` / `已拒绝(rejected)`。合法迁移：不需签批者落库即 `生效`；需签批者落库为 `待签`，`待签 →(QA 签批) 生效`、`待签 →(QA 拒绝) 已拒绝`、`生效 →(增量重算取代) 已取代`。`已取代` 与 `已拒绝` 为终态。
- Q: QA 复核员拒绝一条待签结论时的行为？ → A: 结论转入 `已拒绝` 终态，其全部被抑制动作作废，需重新评估产生新结论；拒绝事件入审计链。
- Q: 结论被增量重算取代时，其已解抑、进入待派发/人工流转中的操作动作如何处置？ → A: 随取代自动作废/撤销（避免按过时结论继续执行操作），作废事件入审计链。
- Q: 自动重算对"待签"结论的作用范围？ → A: 仅重算生效结论；待签结论不被自动重算覆盖（保留待签事实，等待签批）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 评估即落库，流水线自举（Priority: P1）

资深分析师对某共线生产场景发起风险评估。系统在返回结论的同时，将该结论**持久化为一个有状态、可寻址的工作流对象**（拥有唯一标识与初始生命周期状态），并据结论结果立即编排其隐含的操作动作。此后该结论可被复核、QA 签批、动作派发与报告导出**直接引用**——整条流水线无需任何人工补数据即可从这条结论开始运转。

**Why this priority**: 这是设计文档标注的 **G1 核心缺口**，也是整条结论流水线的**唯一源头**。没有"第一条落库结论"，G2 的待签闸门、增量重算、动作签批、报告导出全部无对象可作用。补齐它即让"评估 → 复核/签批 → 动作 → 报告"端到端成立，是其余故事的前提，且独立可交付价值。

**Independent Test**: 在干净环境（无任何预置结论）下发起一次评估，验证：①返回结论的同时数据中存在一条可按标识检索、带初始状态的结论；②该结论的隐含操作动作已被编排登记；③紧接着可对其执行复核/签批/报告导出而无需任何额外补数据步骤。

**Acceptance Scenarios**:

1. **Given** 系统中无任何已落库结论，**When** 分析师对一个场景发起评估，**Then** 系统持久化一条带唯一标识与初始生命周期状态的结论，并返回其标识。
2. **Given** 一条刚落库的结论，**When** 查询该结论，**Then** 返回其当前生命周期状态、结论结果与（如有）已编排的操作动作清单。
3. **Given** 评估结果隐含需要专用化/灭活/再清洁等操作，**When** 结论落库，**Then** 对应操作动作被编排登记，但在结论尚未生效前不对外派发（处于被抑制态）。
4. **Given** 一条已落库并生效的结论，**When** 请求导出其风险评估报告，**Then** 报告成功生成，无需任何中间补数据。

---

### User Story 2 - 高风险结论自动进入 QA 待签闸门（Priority: P2）

当评估产出**高风险结论**（满足既定判据：高总体风险等级、需专用化的结果、或命中青霉素/头孢/高致敏等高危场景之一）时，系统自动将该结论置入"待 QA 签批"态：其隐含的对外操作动作被**全部抑制**，不对外派发；直至一名 QA 复核员完成符合 Part 11 的电子签名，结论方才生效、被抑制的动作随之解除并进入可派发态。低风险（不满足判据）的结论则在落库时即刻生效，直发后续动作。

**Why this priority**: 这是设计文档标注的 **G2 缺口**，直接关系合规闸门是否真实生效。在 P1 让结论可落库后，必须保证高风险结论**不会绕过 QA 签批**就触发对外动作——这是制药合规（Part 11）与共线用药安全的硬要求。依赖 P1（先有落库结论才能 arm 闸门）。

**Independent Test**: 构造一条命中高风险判据的结论与一条不命中的结论，验证：①前者自动进入待签态且零对外动作派发；②后者落库即生效；③对前者完成 QA 电子签名后其转为生效、被抑制动作解除为待派发；④缺少有效签名时无法令其生效。

**Acceptance Scenarios**:

1. **Given** 评估结果命中高风险判据，**When** 结论落库，**Then** 结论自动进入"待签"态，且其全部隐含操作动作处于被抑制态、零对外派发。
2. **Given** 评估结果未命中任何高风险判据，**When** 结论落库，**Then** 结论即刻生效，其操作动作进入可派发态。
3. **Given** 一条待签结论，**When** QA 复核员提交带重认证与签名含义的有效电子签名，**Then** 结论转为生效、签名不可分割地绑定到该结论、被抑制动作解除为待派发，且该签批事件入审计链。
4. **Given** 一条待签结论，**When** 在无有效 QA 签名的情况下尝试令其生效或派发其动作，**Then** 系统拒绝并保持结论于待签态、动作仍被抑制。
5. **Given** 一条待签结论，**When** QA 复核员拒绝该结论，**Then** 结论转入 `已拒绝` 终态、其全部被抑制动作作废、该拒绝事件入审计链，且需重新评估方能产生新结论。

---

### User Story 3 - 事实变更自动召回重算（近实时）（Priority: P3）

上游事实发生变更（如排期、设备或物料状态改变）时，系统**自动**召回受影响且当前生效的结论并对相交子图增量重算，无需任何人工触发。重算产出刷新后的结论，旧结论被置为失效并链接到其替代结论（取代链得以保留），刷新结论再据其结果重新编排动作。整个"事实变更 → 结论刷新"在近实时窗口内完成。

**Why this priority**: 这是设计文档标注的 **G3 缺口**——物化与重算之间断链，使 R-WF8 实为半自动。补齐订阅桥接后，平台才能"事实一变、结论自动跟随"，达成近实时事实源推理的承诺。依赖 P1（需有生效结论作为重算对象）。

**Independent Test**: 在存在若干生效结论的环境中注入一次与部分结论子图相交的事实变更，验证：①无任何人工触发的前提下，相交且生效的结论被自动重算；②不相交或已失效的结论不被触动；③产生刷新结论时旧结论被置失效并链接到替代；④端到端时延落在既定近实时窗口内。

**Acceptance Scenarios**:

1. **Given** 系统中存在与某事实相交的生效结论，**When** 该事实发生变更，**Then** 系统在无人工触发下自动对相交子图重算并产出刷新结论。
2. **Given** 一次事实变更，**When** 自动重算执行，**Then** 仅作用于子图相交且当前生效的结论，不相交或已失效者不受影响。
3. **Given** 重算产出一条刷新结论，**When** 刷新完成，**Then** 旧结论被置为 `已取代` 并链接到其替代结论（取代链可追溯），刷新结论据结果重新编排动作。
4. **Given** 被取代的旧结论此前已有解抑、进入待派发/人工流转中的操作动作，**When** 取代发生，**Then** 这些未完成的旧动作随取代自动作废/撤销，且作废事件入审计链。
5. **Given** 一次事实变更进入系统，**When** 自动链路运行，**Then** 从变更到结论刷新的时延落在既定近实时窗口（≤ 5 秒）内。

---

### User Story 4 - 显式生命周期与集中守卫（Priority: P4）

每条结论携带一个取自**既定枚举集合**的显式生命周期状态；任何状态迁移都经由**单一的迁移合法性来源**一致校验，无论从哪个入口发起。非法迁移（如越过待签直接生效、对已失效结论再签批、operator 触发流转）被一致拒绝并给出可操作的原因，且拒绝时**不改变**任何状态。

**Why this priority**: 这是设计文档 §6 路线 A 的"硬化"部分——把分散在三字段、各处 `if` 自守的隐式状态固化为显式、可校验的状态机。它把 P1–P3 建立的行为沉淀为**可审查、防越权跃迁**的治理层，提升合规可信度。优先级最低，因 P1–P3 的功能闭环可先于此交付，但本故事是质量与合规的收口。

**Independent Test**: 枚举若干合法与非法迁移序列，验证：①合法序列按既定状态集推进；②每一类非法迁移均被拒绝、状态不变、并附明确原因；③相同的迁移规则在不同入口（评估落库、签批、动作流转、增量重算）下表现一致。

**Acceptance Scenarios**:

1. **Given** 一条处于某生命周期状态的结论，**When** 发起一个该状态下合法的迁移，**Then** 结论推进到目标状态并记录该迁移于审计链。
2. **Given** 一条结论，**When** 发起一个非法迁移（违反 from-状态前置），**Then** 系统拒绝、保持原状态不变、并返回说明被拒原因的信息。
3. **Given** 同一类迁移可由多个入口发起，**When** 分别从不同入口发起，**Then** 迁移合法性判定结果一致（不存在某入口有守卫、另一入口无守卫的情况）。
4. **Given** 一名 operator（只读角色），**When** 其尝试推进任一结论状态或签批，**Then** 系统拒绝。

---

### Edge Cases

- **重复评估**：对相同输入重复发起评估，默认各自产生一条独立结论（不做去重/幂等合并）；两条结论各自独立走生命周期。
- **签批竞态**：对一条**已被增量重算取代（`已取代`）**的待签结论尝试签批——系统应拒绝并提示该结论已被取代，引导对最新替代结论操作。
- **重算遇待签**：事实变更命中一条**待签**结论时——以"仅重算生效结论"为界，待签结论**不**被自动重算覆盖（保留待签事实，等待签批），不被刷新或取代。
- **QA 拒绝**：QA 复核员拒绝一条待签结论——结论转入 `已拒绝` 终态、其全部被抑制动作作废、需重新评估方能产生新结论；对 `已拒绝` 终态结论再签批/再拒绝应被拒绝。
- **取代作废动作**：被取代的旧结论若已有解抑、流转中的操作动作——这些未完成动作随取代自动作废/撤销，不得按过时结论继续执行。
- **空变更重算**：重算产出与旧结论**实质等价**的结果时，是否仍生成新结论并取代——默认仍走刷新+取代以保留事实流痕迹，但不得重复派发等价对外动作。
- **动作早派发防护**：在结论尚未生效（待签或被抑制）时请求派发其操作动作——必须被拒绝/抑制。
- **审计链断裂**：任一工作流事件（落库/迁移/签批/派发/取代）缺失或被篡改时，链验真应能定位首个断裂点。
- **建议性回写被拒**：对外建议性回写返回"不接受（not accepted）"时，**不**视为工作流失败（建议性，非权威写）。
- **角色缺失/越权**：身份头缺失或角色不足时，写/迁移/签批端点一致拒绝。

## Requirements *(mandatory)*

### Functional Requirements

**结论持久化与自举（G1）**

- **FR-001**: 系统 MUST 在执行评估时，将每条风险评估结论持久化为带**唯一标识**与**初始生命周期状态**的有状态工作流对象，供下游复核、签批、动作派发与报告导出引用。
- **FR-002**: 系统 MUST 允许按标识检索任一结论，并返回其当前生命周期状态、结论结果与（如有）已编排的操作动作清单。
- **FR-003**: 系统 MUST 在结论落库时据其结果编排隐含的操作动作；并 MUST NOT 在结论尚未生效前对外派发任何操作动作（保持被抑制态）。
- **FR-004**: 系统 MUST 使整条工作流可自举——在无任何预置结论与无人工补数据的前提下，由一次评估即可启动"评估 → 复核/签批 → 动作 → 报告"全链路。

**强制 QA 闸门（G2）**

- **FR-005**: 系统 MUST 在结论落库时据风险特征自动判定其是否需要 QA 签批，判据至少包含：高总体风险等级、需专用化的结果、以及指定高危场景（青霉素/头孢/高致敏等）。
- **FR-006**: 需 QA 签批的结论 MUST 进入"待签"态，且其全部隐含操作动作 MUST 被抑制（不对外派发）。
- **FR-007**: 不需 QA 签批的结论 MUST 在落库时即刻生效，其操作动作进入可派发态。
- **FR-008**: 系统 MUST 要求一枚有效的 QA 电子签名（含重认证与记录的签名含义）方可令待签结论生效；签名生效后，该结论被抑制的动作 MUST 解除为可派发态，且签名 MUST 不可分割地绑定到该结论。
- **FR-009**: 系统 MUST NOT 允许任何未经有效 QA 签名的途径令待签结论生效或派发其被抑制动作。

**事实变更自动重算（G3）**

- **FR-010**: 系统 MUST 在上游事实变更时**自动**（无人工触发）召回并重算受影响的结论。
- **FR-011**: 自动重算 MUST 仅作用于子图与变更事实相交、且当前 `生效` 的结论；不相交、已失效或处于 `待签` 态的结论 MUST NOT 被自动重算覆盖（待签结论保留其待签事实，等待签批，不被刷新或取代）。
- **FR-012**: 当重算产出刷新结论时，系统 MUST 将旧结论置为 `已取代` 并链接到其替代结论，保留可追溯的取代链；刷新结论 MUST 据其结果重新编排动作。被取代旧结论中已解抑、未完成（待派发/流转中）的操作动作 MUST 随取代自动作废/撤销，且作废事件 MUST 入审计链。
- **FR-013**: 系统 MUST 使"事实变更 → 结论刷新"端到端时延落在既定近实时窗口（≤ 5 秒）内。

**显式生命周期与集中守卫（路线 A 硬化）**

- **FR-014**: 每条结论 MUST 携带取自**既定四态枚举集合**的显式生命周期状态：`待签(pending_signature)` / `生效(effective)` / `已取代(superseded)` / `已拒绝(rejected)`，其中 `已取代` 与 `已拒绝` 为终态。
- **FR-015**: 系统 MUST 对每一次状态迁移依据**单一的迁移合法性来源**进行校验，无论迁移由哪个入口发起，判定结果 MUST 一致。合法迁移集为：落库→`生效`（不需签批）、落库→`待签`（需签批）、`待签`→`生效`（QA 签批）、`待签`→`已拒绝`（QA 拒绝）、`生效`→`已取代`（增量重算取代）。
- **FR-016**: 系统 MUST 拒绝任何非法迁移（违反 from-状态前置或自终态外迁），返回可操作的被拒原因，且拒绝时 MUST NOT 改变任何状态。

**角色门禁与审计（宪章对齐）**

- **FR-017**: 系统 MUST 实施角色门禁：仅资深分析师可发起评估与推进结论流转；仅 QA 可施加 QA 签名；operator 对结论与动作 MUST 为只读。
- **FR-018**: 系统 MUST 将每一次结论落库、状态迁移、签批、动作派发与取代记录入**防篡改、可验真、追加式**的审计链；链验真 MUST 能定位首个断裂点。
- **FR-019**: 对外"建议性回写"返回"不接受（not accepted）"时，系统 MUST NOT 将其视为工作流失败（建议性，非权威写）。

**QA 拒绝路径**

- **FR-020**: 系统 MUST 支持 QA 复核员拒绝一条 `待签` 结论：拒绝后该结论 MUST 转入 `已拒绝` 终态、其全部被抑制操作动作 MUST 作废、拒绝事件（含签名人/角色/时间）MUST 入审计链；`已拒绝` 为终态，MUST NOT 再被签批或令其生效，纠正路径为重新发起评估产生新结论。

### Key Entities

- **风险评估结论（Conclusion）**：一次推理产出的结论及其合规/实时状态载体；拥有唯一标识、生命周期状态、结论结果、受影响子图、（如需）签名绑定与取代链接。是整条工作流的核心对象。
- **生命周期状态（Lifecycle State）**：结论所处阶段的显式四态枚举值——`待签(pending_signature)` / `生效(effective)` / `已取代(superseded)` / `已拒绝(rejected)`（后两者为终态），决定允许的迁移与对外动作是否可派发。
- **QA 电子签名（Electronic Signature）**：QA 对结论的不可抵赖签批记录，含签名人、角色、含义、重认证标记与时间；绑定到具体结论，是高风险结论生效的闸门。
- **操作动作（Operational Action）**：结论隐含的内部工单/任务（专用化工单、灭活/再清洁任务、排期阻断、告警、建议性回写等），具被抑制 → 待派发 → 流转的生命周期。
- **事实变更事件（Fact-Change Event）**：上游事实源变更的通知，携带受影响子图，触发对相交生效结论的自动重算。
- **取代链接（Supersession Link）**：旧结论指向其刷新替代结论的链接，保留结论演进的可追溯历史。
- **审计链条目（Audit Trail Entry）**：对每个工作流事件的追加式、防篡改记录，支撑全程可追溯与验真。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 分析师发起的每一次评估都产出一条可检索、带初始状态的持久化结论——评估后结论丢失率为 **0%**。
- **SC-002**: 在干净环境（零预置结论、零人工补数据）下，可由单次评估端到端走通"评估 → 复核/签批 → 动作 → 报告"全链路（**自举成功**）。
- **SC-003**: 满足高风险判据的结论 **100%** 自动进入待签态，且在获得有效 QA 签名前对外派发的操作动作数为 **0**。
- **SC-004**: 上游事实变更触发对全部相交生效结论的自动重算，无任何人工步骤，端到端时延 **≤ 5 秒**；不相交/已失效结论被误重算的比例为 **0%**。
- **SC-005**: 非法生命周期迁移被拒绝的比例为 **100%**，每次拒绝均不改变状态并附明确原因。
- **SC-006**: 全部工作流事件在审计链中**齐备且可验真**；对任一人为篡改/缺失，链验真 **100%** 能定位首个断裂点。
- **SC-007**: operator 角色对结论的写/迁移/签批尝试被拦截的比例为 **100%**。

## Assumptions

- **范围采路线 A**：本特性交付 G1/G2/G3 闭环 + 显式生命周期状态机；**不含**路线 B（迁移定义数据化为本体库可治理的 `OntologyAction` / 通用解释执行引擎），其作为后续特性评估。
- **复用推理内核**：沿用既有推理规则、计算器与合规机制（哈希链、QA 签名、Action 编排、增量重算）不改写；本特性只补"接通源头"与"显式化状态"两层，不引入新风险规则。
- **近实时阈值**：端到端"事实变更 → 结论刷新"目标时延取 **≤ 5 秒**（沿用 002 的近实时口径）。
- **高风险判据初始集**：高总体风险等级 OR 需专用化 OR 命中青霉素/头孢/高致敏高危场景；判据可在后续迭代细化，本特性以该初始集为准。
- **签名合规强度**：QA 电子签名为 **21 CFR Part 11** 级（重认证 + 记录含义/时间/签名人 + 不可分割绑定到结论）。
- **审计为全局单链**：沿用 002 的全局追加式哈希链作为审计机制（非每实例链）。
- **重复评估非幂等**：对相同输入的重复评估默认各自落库为独立结论；幂等/去重不在本特性范围。
- **对外回写为建议性**：对上游系统（如排期）的回写为建议性、非权威；"不接受"非失败。
- **身份经网关注入**：用户身份/角色经可信网关注入（`X-User`/`X-Role`），SSO 可插拔、后续接入。

## Dependencies

- 依赖 002 已交付的结论/动作/签名模型、全局哈希链审计、Action 编排引擎、增量重算与取代链、QA 签名与报告导出能力——本特性在其上补齐自举源头与显式状态治理。
- 依赖至少一个真实事实源（002 引入的生产排期 APS 连接器）产生携子图的事实变更事件，以驱动 G3 的自动重算。
