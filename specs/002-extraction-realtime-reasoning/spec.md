# Feature Specification: 多源抽取与实时推理 —— 能力二/能力三 GAP 闭合

**Feature Branch**: `002-extraction-realtime-reasoning`

**Created**: 2026-06-22

**Status**: Draft

**Input**: User description: "@docs/gap-analysis.md 实现能力二和能力三GAP"

## Overview

能力一（知识模型 / T-Box 维护工作台）已全栈落地并无 gap。本特性闭合 **能力二（多源抽取与实体对齐）** 与 **能力三（实时事实源对齐与 Action 推理）** 的剩余缺口——即 `docs/gap-analysis.md` 对应方案路线图的 **P2–P5**：

- **能力二**：抽取流水线已实现但**未接线**（`create_job` 仅写 `pending`、`run_extraction_pipeline` 为零调用方的死代码）；数据库源读取器缺失；对齐审核 UI 仅为静态占位页。本特性把流水线接通、补齐数据库源读取、并建成人工对齐审核闭环。
- **能力三**：推理内核（OWL 结构推理 + 4 组规则 + MACO/PDE 计算 + 冲突裁决）已完备且无 gap；但事实源连接器仅为 Stub、无增量物化与事实变更事件、无 Action 引擎、无实时看板、无风险报告导出。本特性接入至少一个真实事实源、建立增量物化与事件驱动重算、落地 Action 编排引擎与风险评估报告，并交付实时推理看板。
- **能力三耦合的合规硬化**：审计哈希链（ALCOA+ 防篡改）、对高风险/专用化/合规阻断类**推理结论**的 QA 强制电子签名、operator/QA 业务流的角色门禁——这些在能力一规范（spec 001）中已明确标注"随能力三作为后续特性交付"，故纳入本特性。

> **现状基线（经 gap-analysis 逐条核验）**：抽取的解析器（Excel/Word）、`llm_extractor`、`aligner.align_entity`、`pipeline.run_extraction_pipeline` 后端代码已存在；推理引擎 `reasoning/engine.py`、规则、计算器、冲突裁决、`/api/reasoning/*` 端点可用；`integration/base.py` 仅有 `ExternalSystemConnector(ABC)` 与 `StubConnector`；`audit_log` 表已建但无哈希链；RBAC 仅覆盖 senior_analyst 写门禁。本规范以"激活已有死代码 + 补齐缺失链路"为交付主线。

## Clarifications

### Session 2026-06-22

- Q: 本特性的交付范围？ → A: 闭合**能力二 + 能力三**全部 gap（含能力三耦合的合规硬化：审计哈希链、推理结论 QA 强制电子签名、operator/QA 角色门禁）；不含企业 SSO 对接（可插拔，后续）。能力一无 gap，不在范围内。
- Q: 首个真实事实源连接器选哪个？ → A: **生产排期（APS）**——作用面最大（设备×产品×时段排产是相容性判定的事实基础），作为首个真实连接器替换 Stub；ERP/MES/LIMS/CTMS 沿用同一连接器抽象，按后续增量接入。
- Q: 排期冲突时对外部排期系统的"回写"性质？ → A: **建议性回写（advisory）**——系统标记冲突、阻断该排期并回写**调整建议**，不直接改写外部排期系统的权威数据；是否采纳由排期方决定。
- Q: 增量重算的触发与时延目标？ → A: **事实变更事件驱动**——仅对受影响子图（相关设备/产品/区域）增量重算，事实变更到结论刷新目标时延 **≤ 5 秒**。
- Q: 事实源增量如何进入平台？ → A: **混合**——以短周期轮询（poll）为默认接入方式，对支持推送的源可选 push（Webhook/消息队列）；轮询周期默认 ≤ 2 秒，为 ≤ 5 秒端到端时延目标留出物化与重算余量。
- Q: 高风险/专用化结论的 QA 电子签名合规强度？ → A: **21 CFR Part 11 级**——签名时重新认证（用户名+密码）、记录签名含义（meaning）、时间、签名人，并将签名不可分割地绑定到对应结论记录；签名为防抵赖。
- Q: Action 引擎产出的工单/任务落地到哪里？ → A: **平台内部记录**（`action_execution` 表）——专用化工单、灭活/再清洁任务以内部记录生成并可人工流转状态；对外仅产生告警与对 APS 的**建议性回写**，不直接在外部系统创建工单。
- Q: 风险评估报告以什么形式交付？ → A: **PDF + 结构化数据双产物**——生成可下载/归档的 PDF（含 QA 签批信息）并同时产出结构化数据（JSON）供前端渲染与程序化复用。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 多源抽取作业接线与进度反馈（Priority: P1）

分析师在抽取工作台上传设备台账 Excel 或工艺规程/SOP Word，创建抽取作业。系统**实际触发**抽取流水线（解析 → 抽取 → 对齐 → 入审核队列），而非仅记录一个 `pending` 作业；作业各阶段进度通过实时反馈对用户可见。表格行被抽为实体实例，正文中的"若…则…必须…"逻辑被抽为带前置/后置条件的 Action 候选；抽取提示注入领域受控词表（OEB/PDE/材质/洁净级别）以降低自由文本错误。当智能抽取能力不可用时，系统回退到结构化原始数据抽取而非失败。

**Why this priority**: 这是 gap-analysis 标注的 **P1 核心缺口**——pipeline 已实现，只差"一根线"（`create_job → run_extraction_pipeline`）。接通后立即激活已有死代码，让"上传文档后真正发生抽取"成立，是能力二其余环节（审核闭环）的前提，且独立可交付价值。

**Independent Test**: 仅交付本故事即可独立测试：上传一份设备台账 Excel 与一份含表格与正文的 SOP Word，创建作业后系统真实执行流水线并产出候选，作业进度（各阶段状态）实时可见；中途模拟智能抽取不可用时，作业回退到结构化抽取并完成而非失败。

**Acceptance Scenarios**:

1. **Given** 一份设备台账 Excel，**When** 创建抽取作业，**Then** 系统实际触发抽取流水线（解析→抽取→对齐→审核队列）并通过进度反馈实时展示各阶段进展，作业最终进入"待审核"而非停留在 `pending`。
2. **Given** 一份含表格与正文的 SOP Word，**When** 抽取完成，**Then** 表格行被抽为实例、正文中的"若…则…必须…"逻辑被抽为带前置/后置条件的 Action 候选。
3. **Given** 抽取目标类体系，**When** 抽取运行，**Then** 抽取提示注入领域受控词表（OEB/PDE/材质/洁净级别）以降低自由文本错误。
4. **Given** 用于逻辑抽取的智能能力不可用，**When** 运行抽取作业，**Then** 系统回退到结构化原始数据抽取并完成作业，记录降级原因而非失败。

### User Story 2 - 跨源实体对齐与人工审核闭环（Priority: P2）

分析师在对齐审核界面查看抽取产生的候选：系统将抽取目标对齐到领域本体类体系，对跨源描述同一对象（同一设备在 Excel/MES/ERP 中的不同标识、同一药品在多源的不同命名）的实例进行对齐归组并标记规范实例（设备以唯一编号为强主键，药品以活性成分+剂型+规格为对齐键）。低置信候选进入审核队列，分析师可对候选执行确认、拒绝、手动合并或拆分，确认后方进入知识库。系统亦支持从数据库源抽取（表结构→实体、外键→关系）。

**Why this priority**: gap-analysis 标注 **P2**——后端 `aligner.align_entity` 已就绪，主要缺口是前端审核 UI（`alignment-review` 组件不存在，`extraction/page.tsx` 仅 60 行静态占位）与数据库源读取器（缺失）。补齐后能力二闭环。

**Independent Test**: 在 US1 产出候选的基础上，跨源同一设备被正确归为一组并标记规范实例；低置信候选出现在审核队列；分析师对候选执行确认/拒绝/合并/拆分，确认项进入知识库、拒绝项不进入；另对一个数据库源运行抽取，表结构被抽为实体、外键被抽为关系。

**Acceptance Scenarios**:

1. **Given** 同一设备在多源中以不同标识出现，**When** 执行对齐，**Then** 系统以设备唯一编号为强主键将其归为一组并标记规范实例。
2. **Given** 抽取产生的低置信候选，**When** 分析师在审核界面操作，**Then** 可对候选执行确认、拒绝、手动合并或拆分，确认后方进入知识库。
3. **Given** 同一编号在不同源指向疑似不同实体，**When** 对齐判定存在歧义，**Then** 候选进入人工审核而非自动合并。
4. **Given** 一个数据库源（表结构与外键），**When** 创建数据库抽取作业，**Then** 表结构被抽为实体类候选、外键被抽为关系候选并进入同一审核队列。

### User Story 3 - 实时事实源接入与增量物化（Priority: P3）

平台配置一个真实事实源连接器（首发：生产排期 APS），增量拉取事实（某设备时段内排产哪些产品/批次、清洁记录、残留检测结果、临床试验用药批次），将增量归一到本体标识并物化为事实实例（A-Box），发布"事实变更事件"。事实变更事件触发增量推理——系统仅对受影响子图（相关设备/产品/区域）重算，并在 ≤ 5 秒内刷新结论。当事实源不可达或超时时，系统保留上一已知良好状态、记录失败并告警，而非污染知识库。

**Why this priority**: gap-analysis 标注 **P3**——这是能力三实时链路的起点。`integration/base.py` 仅有 Stub，无真实连接器、无 `fact_materialization_run`、无事件机制。先做 APS（作用面最大）替换 Stub，建立增量物化与事件驱动重算。

**Independent Test**: 配置 APS 连接器并注入一次事实变更（设备排产组合变化），系统将增量归一为事实实例、发布事实变更事件、仅对受影响子图增量重算并在 ≤ 5 秒内刷新结论；随后模拟事实源超时，系统保留上一良好状态并告警，知识库不被污染。

**Acceptance Scenarios**:

1. **Given** 已配置的 APS 真实连接器，**When** 事实源产生增量变更，**Then** 系统将增量归一到本体标识、物化为事实实例并发布事实变更事件。
2. **Given** 一次事实变更事件，**When** 触发推理，**Then** 系统仅对受影响子图（相关设备/产品/区域）重算，并在不超过 5 秒内刷新结论。
3. **Given** 事实源不可达或超时，**When** 同步执行，**Then** 系统保留上一已知良好状态、记录失败并告警，不污染知识库。
4. **Given** 一次增量物化运行，**When** 事后追溯，**Then** 可查到该次物化的来源、水位、变更条目与产生的事实变更事件记录。

### User Story 4 - Action 编排引擎与风险评估报告（Priority: P4）

当增量推理得出特定结论时，系统自动编排相应动作：得出"需专用化"结论时生成专用化工单与告警；得出需灭活/再清洁结论时创建灭活验证或清洁再验证任务；当排期使不相容产品同设备同时段时，标记冲突、阻断该排期并向排期系统**回写调整建议**（建议性，不直接改写外部权威数据）；并可生成风险评估报告（RiskAssessmentReport）作为产物。每次动作执行均记录留痕（动作类型、触发结论、规则链、执行结果、回写状态）。

**Why this priority**: gap-analysis 标注 **P4**——依赖 P3 的事实流。全库 grep `RaiseDedicationRequirement`/`ActionExecution`/`action_engine` 零命中、无 `action_execution` 表、无报告导出器。本故事把"推理结论"转化为"可执行动作 + 可交付产物"。

**Independent Test**: 在 US3 事实流的基础上，注入一组触发"需专用化"的事实，系统自动生成专用化工单与告警并记录动作执行留痕；注入一组不相容同设备同时段排期，系统阻断并回写调整建议；对一次完成的评估生成风险评估报告产物并可下载/查看。

**Acceptance Scenarios**:

1. **Given** 推理得出"需专用化"结论，**When** 动作编排执行，**Then** 系统生成专用化工单与告警，并记录动作执行留痕。
2. **Given** 推理得出需灭活/再清洁结论，**When** 动作编排执行，**Then** 系统创建灭活验证或清洁再验证任务并留痕。
3. **Given** 排期使不相容产品同设备同时段，**When** 检测到冲突，**Then** 系统标记冲突、阻断该排期并向排期系统回写**调整建议**（不直接改写外部权威数据）。
4. **Given** 一次完成的评估，**When** 请求生成风险评估报告，**Then** 系统产出含分类/专用化决策/污染途径评分/CFDI 情景/MACO·PDE 数值/规则链与法规依据的报告产物。
5. **Given** 任一动作执行，**When** 事后追溯，**Then** 可查到动作类型、触发结论、规则链、执行结果与回写状态的完整留痕。

### User Story 5 - 实时推理看板（Priority: P5）

操作员或分析师打开实时推理看板，查看设备×产品的共线相容性热力图、未来若干天的排期风险，并可展开任一结论的推理链溯源（触发规则 ID 链与对应法规依据）。看板在事实变更后近实时刷新（与 US3 的 ≤ 5 秒目标一致）。

**Why this priority**: gap-analysis 标注 **P5**——`realtime-inference-panel`/`compatibility-matrix` 组件零命中，`integration/page.tsx` 仅 63 行只读 specs 展示。看板是能力三的"可视化闭环"，依赖 P3/P4 的事实流与结论/动作。

**Independent Test**: 在 US3/US4 基础上，看板展示设备×产品相容性热力图与未来排期风险；触发一次事实变更后看板近实时刷新；点击任一结论可展开其规则链 ID 与法规依据。

**Acceptance Scenarios**:

1. **Given** 已有事实与推理结论，**When** 用户打开实时看板，**Then** 看板展示设备×产品的共线相容性热力图与未来若干天排期风险。
2. **Given** 一次事实变更事件，**When** 看板订阅更新，**Then** 看板在近实时（≤ 5 秒）内刷新对应单元。
3. **Given** 任一已产出结论，**When** 用户在看板上展开该结论，**Then** 展示其触发规则 ID 链及对应法规依据（推理链溯源）。

### User Story 6 - 合规硬化：审计哈希链、QA 强制电子签名与角色门禁（Priority: P5）

平台对全链路（抽取/对齐/物化/推理/动作）操作的审计记录实现**只追加且具完整性保护的哈希链**（每条记录链接前一条的哈希，篡改可检测）。对高风险/专用化/合规阻断类**推理结论**强制 QA 电子签名后方可生效——未签名前结论不生效、不触发对外动作。按角色分级授权扩展到 operator/QA 业务流（高级分析师维护知识模型与发布、操作员查看推理、QA 复核高风险结论）。

**Why this priority**: gap-analysis 标注 **P5（合规硬化）**——`audit_log` 表已建但 grep `prev_hash`/`hash_chain`/`sha256` 零命中；无电子签名流程；`require_role` 仅覆盖 ontology 写接口。spec 001 已将 FR-034（QA 强制电子签名）与 ALCOA+ 哈希链明确标注"随能力三作为后续特性交付"，故归入本特性。

**Independent Test**: 对一条高风险/专用化结论验证"未经 QA 电子签名不生效、不触发对外动作"，QA 签名后方生效；对一段审计记录验证哈希链完整性校验通过，且人为篡改某条记录后校验报告检测到断链；以 operator/QA 角色登录验证各自权限边界。

**Acceptance Scenarios**:

1. **Given** 任一抽取/对齐/物化/推理/动作操作，**When** 操作发生，**Then** 系统写入只追加且链接前一条哈希的审计记录，完整性校验可验证未被篡改。
2. **Given** 一条已写入的审计记录被人为修改，**When** 运行完整性校验，**Then** 系统检测到哈希断链并定位被篡改位置。
3. **Given** 一条高风险/专用化/合规阻断结论，**When** QA 未电子签名，**Then** 结论不生效且不触发对外动作；QA 电子签名后方生效。
4. **Given** operator 与 QA 角色，**When** 各自登录操作，**Then** 系统按角色限制可执行的操作范围（operator 只读推理、QA 可复核签名、维护/发布仍限高级分析师）。

### Edge Cases

- **抽取无外部模型能力**：用于逻辑抽取的智能能力不可用时，系统回退到结构化原始数据抽取而非失败，并记录降级原因。
- **跨源对齐歧义**：同一编号在不同源指向疑似不同实体时，进入人工审核而非自动合并。
- **事实源不可达/超时**：连接器无法连接或超时时，保留上一已知良好状态、记录失败并告警，而非污染知识库。
- **重复/乱序事实事件**：同一事实变更被重复投递或乱序到达时，物化须幂等且以水位/版本去重，避免重复物化或回退覆盖新值。
- **增量重算风暴**：短时间大量事实变更时，受影响子图重算须可合并/限流，避免推理积压突破 ≤ 5 秒目标（必要时降级为批量刷新并标注）。
- **规则冲突**：多条规则对同一对象给出相反的专用化结论时，以安全优先（"需专用化"优先）裁决，保留冲突双方与裁决依据可追溯（沿用已具备的冲突裁决能力）。
- **缺失关键数据**：PDE 等关键数据缺失时，MACO 计算回退到可用方法（剂量法/通用限/NOEL 法）并标注所用方法与数据缺口（沿用已具备的计算能力）。
- **动作回写被拒**：外部排期系统拒绝或未采纳回写建议时，系统记录回写状态为"已建议/未采纳"，不视为失败，结论与告警仍保留。
- **QA 签名缺位下的动作触发**：高风险结论在 QA 未签名前，对外动作（工单/告警/回写）须被抑制并置于待签名状态，签名后方可触发。
- **审计链中断恢复**：哈希链校验发现断点时，系统须能定位首个断裂记录并告警，不得静默续写掩盖断链。

## Requirements *(mandatory)*

### Functional Requirements

#### 能力二：多源抽取与实体对齐

- **FR-001**: 系统 MUST 在创建抽取作业时**实际触发**抽取流水线（解析 → 抽取 → 对齐 → 审核队列），作业不得停留在仅记录的 `pending` 状态。
- **FR-002**: 系统 MUST 提供抽取作业的实时进度反馈，逐阶段展示进展与最终状态。
- **FR-003**: 系统 MUST 支持从 Excel（设备台账/物料/批记录）抽取实体、属性与实例。
- **FR-004**: 系统 MUST 支持从 Word（SOP/工艺规程）抽取实体/实例（表格行）与过程实体/Action（正文逻辑）。
- **FR-005**: 系统 MUST 将"若…则…必须…"类逻辑转化为带前置/后置条件的 Action 候选。
- **FR-006**: 系统 MUST 在抽取过程中注入领域受控词表（OEB/PDE/材质/洁净级别）以降低自由文本错误。
- **FR-007**: 系统 MUST 在用于逻辑抽取的智能能力不可用时回退到结构化原始数据抽取，记录降级原因而非失败。
- **FR-008**: 系统 MUST 将抽取目标对齐到领域本体类体系。
- **FR-009**: 系统 MUST 对跨源描述同一对象的实例进行对齐归组并标记规范实例（设备以唯一编号为强主键，药品以活性成分+剂型+规格为对齐键）。
- **FR-010**: 系统 MUST 将低置信抽取/对齐结果送入人工审核队列；分析师 MUST 能对候选执行确认、拒绝、手动合并或拆分，确认后方进入知识库。
- **FR-011**: 系统 MUST 在对齐判定存在歧义（同一编号疑似指向不同实体）时进入人工审核而非自动合并。
- **FR-012**: 系统 MUST 支持从数据库源抽取（表结构 → 实体类候选、外键 → 关系候选），并送入同一审核队列。
- **FR-013**: 系统 MUST 提供对齐审核界面，展示候选、置信度、跨源归组与规范实例标记，并承载确认/拒绝/合并/拆分操作。

#### 能力三：实时事实源对齐、Action 推理与看板

- **FR-014**: 系统 MUST 提供至少一个真实事实源连接器（首发：生产排期 APS），替换 Stub；连接器 MUST 复用统一的外部系统连接抽象以便后续接入 ERP/MES/LIMS/CTMS。接入方式以**短周期轮询（默认 ≤ 2 秒）**为主，并 MUST 支持对可推送源的可选 push（Webhook/消息队列）。
- **FR-015**: 系统 MUST 增量拉取事实并将其归一到本体标识、物化为事实实例（A-Box）。
- **FR-016**: 系统 MUST 在事实物化后发布事实变更事件，并保留每次物化运行的来源/水位/变更条目/事件记录以供追溯。
- **FR-017**: 系统 MUST 在事实变更后仅对受影响子图（相关设备/产品/区域）进行增量重算。
- **FR-018**: 系统 MUST 在事实源不可达或超时时保留上一已知良好状态、记录失败并告警，不污染知识库。
- **FR-019**: 系统 MUST 保证事实物化对重复/乱序事件幂等（以水位/版本去重）。
- **FR-020**: 系统 MUST 在推理得出"需专用化"结论时生成专用化工单与告警；工单为**平台内部记录**（可人工流转状态），不直接在外部系统创建。
- **FR-021**: 系统 MUST 在推理得出需灭活/再清洁结论时创建灭活验证或清洁再验证任务（同为平台内部记录）。
- **FR-022**: 系统 MUST 在排期使不相容产品同设备同时段时标记冲突、阻断该排期并向排期系统回写**调整建议**（建议性，不直接改写外部权威数据）。
- **FR-023**: 系统 MUST 为每次动作执行记录留痕（动作类型、触发结论、规则链、执行结果、回写状态）。
- **FR-024**: 系统 MUST 支持生成风险评估报告（RiskAssessmentReport），含分类结果、专用化决策、污染途径评分、CFDI 情景、MACO/PDE 数值、规则链与法规依据；交付为 **PDF（可下载/归档，含 QA 签批信息）+ 结构化数据（JSON）双产物**。
- **FR-025**: 系统 MUST 提供实时推理看板，含设备×产品共线相容性热力图、未来若干天排期风险与推理链溯源。
- **FR-026**: 看板 MUST 在事实变更后近实时刷新（与 ≤ 5 秒时延目标一致）。
- **FR-027**: 系统 MUST 复用已具备的推理内核（OWL 结构推理、药品风险分类、设备专用化决策、污染途径评分、CFDI 情景识别、MACO/PDE 计算、安全优先冲突裁决、规则链溯源），不重复实现。

#### 跨切：合规、审计与权限（能力三耦合）

- **FR-028**: 系统 MUST 对全链路（抽取/对齐/物化/推理/动作）操作写入只追加的审计记录，并以哈希链（每条链接前一条哈希）提供完整性保护。
- **FR-029**: 系统 MUST 提供审计哈希链完整性校验，能检测篡改并定位首个断裂记录。
- **FR-030**: 系统 MUST 对高风险/专用化/合规阻断类推理结论强制 QA 电子签名后方可生效；未签名前结论不生效且不触发对外动作。电子签名 MUST 达到 **21 CFR Part 11 级**：签名时重新认证、记录签名含义/时间/签名人并不可分割地绑定到结论记录。
- **FR-031**: 系统 MUST 将基于角色的访问控制扩展到 operator/QA 业务流（operator 查看推理、QA 复核与签名、维护/发布仍限高级分析师）。

### Key Entities *(include if feature involves data)*

- **抽取作业（Extraction Job）**：抽取配置、源类型（Excel/Word/数据库）、状态机（pending→running→各阶段→待审核→完成/失败）、进度、降级标记。
- **抽取候选（Extraction Candidate）**：抽取出的实体/属性/关系/Action 候选，含置信度、对齐目标本体类、跨源归组、规范实例标记、审核状态（待审核/确认/拒绝/合并/拆分）。
- **数据库源描述**：被抽取数据库的表结构与外键元信息（→ 实体类/关系候选的来源）。
- **事实源连接器配置**：外部系统（首发 APS，后续 ERP/MES/LIMS/CTMS）的接入配置与同步水位。
- **事实物化运行（Fact Materialization Run）**：一次增量物化的来源、水位、变更条目、产生的事实变更事件与结果状态。
- **事实实例（A-Box 事实）**：由连接器物化的具体个体（某设备某时段排产组合、批次、清洁/残留记录）。
- **事实变更事件（Fact-Change Event）**：触发增量重算的事件，携带受影响子图标识。
- **风险评估结论**：分类结果、专用化决策、污染途径评分、CFDI 情景、MACO/PDE 数值、规则链与法规依据、冲突裁决记录、生效状态（待 QA 签名/已生效）。
- **动作执行（Action Execution）**：由结论触发的具体动作（专用化工单、灭活/清洁再验证任务、排期阻断+建议回写、报告生成）及其执行/回写留痕。
- **风险评估报告（RiskAssessmentReport）**：可交付的报告产物。
- **审计记录（哈希链）**：全链路操作留痕，只追加并以前后哈希链接提供完整性保护。
- **电子签名**：QA 对高风险结论的签名记录（签名人、时间、对象、生效状态变更）。
- **用户与角色**：高级分析师、操作员（operator）、QA 及其权限边界。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 上传设备台账 Excel 或 SOP Word 创建作业后，系统 100% 实际触发抽取流水线并产出候选（不再停留在 `pending`），作业各阶段进度对用户可见。
- **SC-002**: 设备台账 Excel 与 SOP Word 抽取后，关键字段抽取准确率达到约定阈值（建议 ≥ 90%，最终由验收与业务方确认）。
- **SC-003**: 跨源描述同一设备/药品的实例被正确归组、规范实例标记正确；分析师可对候选执行确认/拒绝/合并/拆分，且仅确认项进入知识库。
- **SC-004**: 至少一个真实事实源（APS）连接器可用并替换 Stub；注入事实变更后，增量归一与物化成功且产生事实变更事件。
- **SC-005**: 事实源发生变更后，相关风险结论与看板刷新时延 ≤ 5 秒（近实时），且仅受影响子图被重算。
- **SC-006**: 推理得出的动作类结论 100% 触发对应 Action 编排（专用化工单/告警、灭活·清洁再验证任务、排期阻断+建议回写、报告生成）并留痕；排期建议为建议性回写，不改写外部权威数据。
- **SC-007**: 每条合规结论 100% 可展开其规则链与对应法规依据（决策可追溯）；可生成风险评估报告产物。
- **SC-008**: 全链路操作 100% 写入只追加审计记录，哈希链完整性校验对未篡改数据通过、对篡改数据检测到断链并定位。
- **SC-009**: 高风险/专用化/合规阻断类结论在 QA 电子签名前 100% 不生效且不触发对外动作，签名后方生效。
- **SC-010**: 角色权限边界生效——operator 仅可查看推理、QA 可复核签名、维护/发布仍限高级分析师；越权操作被拒绝。

## Assumptions

- **范围**：本特性交付**能力二 + 能力三**全部 gap（含能力三耦合的合规硬化：审计哈希链、推理结论 QA 强制电子签名、operator/QA 角色门禁）。**范围外**：能力一（无 gap）；企业 SSO 对接（角色门禁保持可插拔身份层，SSO 后续接入）；方案文档中超出 gap-analysis P2–P5 的全平台愿景仅作上下文。
- **首发连接器**：以生产排期（APS）为首个真实连接器（作用面最大）；ERP/MES/LIMS/CTMS 复用同一连接抽象，按后续增量接入，不在本特性硬性范围内。
- **回写性质**：对外部排期系统的回写为**建议性**（advisory），不直接改写外部权威数据；采纳与否由排期方决定。
- **近实时时延目标**：事实变更到结论/看板刷新的目标时延为 ≤ 5 秒；重算风暴时允许降级为批量刷新并标注。
- **复用既有内核**：抽取的解析器/`llm_extractor`/`aligner`/`pipeline`、推理 `engine`/规则/计算器/冲突裁决/`/api/reasoning/*` 均已存在，本特性以"接线 + 补缺"为主，不重复实现已具备的能力。
- **人在环**：抽取/对齐结果均带置信度须经分析师审核方可发布；高风险合规结论强制 QA 电子签名。
- **冲突解决策略**：多规则冲突时以"需专用化"安全优先，与法规"合规优先"一致（沿用已具备的冲突裁决）。
- **抽取准确率阈值**：未另行约定时取业界常见基线（建议 ≥ 90%），最终阈值由验收与业务方确认。
- **合规框架**：以制药 GMP 数据完整性（ALCOA+）与 CFDI 共线生产监管要求为合规基线。
- **既有基线**：平台已具备只读本体浏览、能力一全栈维护工作台、A-Box 个体管理与可用的规则/数值推理引擎；本特性以闭合抽取接线/对齐审核/实时联动/动作编排/合规硬化等缺口为交付主线。
- **用户群体**：主要用户为制药企业的高级分析师、生产操作员（operator）与 QA 人员，具备相应领域知识。
