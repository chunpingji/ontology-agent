## 1. 方案概述与目标

### 1.1 业务背景

临床药物生产（尤其是**共线生产 Shared-Line Production**）面临交叉污染风险评估的信息集成挑战：药品风险属性、设备 GMP 属性、污染途径、清洁验证、设施环境、8 种 CFDI 特殊情景等知识分散在 SOP、Excel 台账、ERP/MES/LIMS/CTMS 等异构系统中，难以统一推理与实时决策。

SLPRA 本体（已建成 7 个 TTL 模块、约 720 条公理）提供了形式化的领域知识，但**缺少一个让分析师维护、从异构源抽取、并与实时事实源联动推理的工程化平台**。本方案即在 infilake-dw 本体工作台基础上，构建这一平台。

### 1.2 三大核心能力（对应需求）

| 编号 | 需求 | 平台能力 | 主要承载 |
|------|------|---------|---------|
| **能力一** | 高级分析师维护核心药物实体、生产区域和设备实体、生产过程实体和关系定义、映射和抽取 | 本体 Schema 维护工作台（Class/LinkType/Action 的 CRUD + 审核 + 映射） | 第 4 节 |
| **能力二** | 从文档（Excel/Word）、数据库实体、逻辑和 action 抽取，实体对齐 | 多源抽取流水线 + 跨源实体对齐 | 第 5 节 |
| **能力三** | 结合本体 KG 和 Action，实时对齐事实源（排期/ERP/MES/CTMS）进行推理 | 事实源连接器 + 物化 KG + OWL/SWRL 推理 + Action 引擎 | 第 6 节 |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **本体即治理契约** | SLPRA TTL 模块是 Schema 的"真理来源"，平台 Class/LinkType/Action 与之双向映射 |
| **双表示法** | OWL（结构知识，分类推理）+ SWRL/规则（条件推理，专用化决策、风险评分）——沿用 OntoReg 方法论 |
| **人在环（Human-in-the-loop）** | 抽取/对齐结果均带 confidence，分析师审核后方可发布；高风险合规结论强制复核 |
| **事实与定义分离** | 本体定义（T-Box）由分析师维护；事实实例（A-Box）由事实源实时物化，二者通过映射层连接 |
| **安全优先冲突解决** | 多规则冲突时 `requiresDedication=true` 优先（与 CFDI"法规合规优先"一致） |

### 2.1 分层职责

- **表现层**：复用 infilake-dw 既有本体页面（`ontology/[projectId]`），新增"共线风险评估"与"实时推理看板"两个领域专用视图。
- **应用层**：复用既有 `services/ontology/*`（抽取、对齐、图、语义查询），扩展 **推理 & Action 服务**、**事实源连接器**、**Excel/Word 文档读取器**。
- **存储层**：PostgreSQL 存 T-Box 与映射；新增 RDF 三元组存储（GraphDB / Apache Jena Fuseki）承载 A-Box 物化与 OWL/SWRL 推理。

## 3. 本体与知识图谱底座

### 3.1 SLPRA 模块 → 平台 Domain 映射

平台中建立一个 **Domain：`clinical-shared-line-production`**，其下挂 7 个 SLPRA 模块作为 Class/LinkType 命名空间分组：

| SLPRA 模块 | 平台 Class 分组 | 核心 Class（示例） |
|-----------|----------------|------------------|
| M1 药品风险属性 | `drug` | DrugProduct, API, Residue, OEBClassification, InactivationDisposition |
| M2 GMP 设备 | `equipment` | Equipment, EquipmentSurface, ConstructionMaterial, CleanabilityRating |
| M3 污染途径 | `contamination` | ResidueContamination, MechanicalTransfer, AirborneTransmission, ConfusionEvent |
| M4 风险评估 | `risk` | RiskAssessmentProcess, MACOCalculation, 8×SharedLineScenario, CompatibilityAssessment |
| M5 清洁验证 | `cleaning` | CleaningProcess, AnalyticalMethod, CleaningValidationProtocol |
| M6 设施环境 | `facility` | ProductionArea, CleanArea(A/B/C/D), ProductionRoom, Airlock |
| 集成桥接 | `integration` | 跨模块属性 + 外部本体对齐（DrOn/ChEBI/ISA-88/IDMP/BFO） |

### 3.2 双存储一致性

- **T-Box（定义）**：分析师在工作台维护 → 持久化到 PostgreSQL（`OntologyClass/LinkType/Action`）→ **导出器**生成/更新对应 `ontology/slpra/*.ttl`（保持与已有 720 公理一致）。
- **A-Box（事实）**：事实源数据经映射 → 物化为 RDF 三元组写入 Triple Store → 触发推理。
- **同步机制**：TTL 为权威基线；平台启动时**导入** TTL 校验 PostgreSQL 与 TTL 无漂移（diff 报告），分析师变更后**导出**回 TTL 并走 Git 版本管理（`ontology/slpra/`）。

### 3.3 外部本体对齐（复用 `slpra-integration.ttl`）

平台保留已建立的对齐公理：`DrugProduct ⊑ dron:00000005`、`API ⊑ chebi:23367`、`Equipment ⊑ isa88:EquipmentEntity`、`DrugProduct ⊑ idmp:MedicinalProduct` 等，使抽取/映射时可借助外部本体术语消歧。

## 4. 能力一：高级分析师维护核心实体与关系

> **目标**：制药企业高级分析师维护核心药物实体、生产区域和设备实体、生产过程实体和关系定义、映射和抽取。

### 4.1 维护对象（对应 SLPRA）

| 维护对象 | 平台实体 | 对应 SLPRA |
|---------|---------|-----------|
| 核心药物实体 | `OntologyClass` (drug 分组) | DrugProduct 及 11 个子类、API 三型、Residue |
| 生产区域实体 | `OntologyClass` (facility 分组) | ProductionArea/CleanArea(A-D)/Room/Airlock |
| 设备实体 | `OntologyClass` (equipment 分组) | Equipment 树、材质、表面、可清洁性 |
| 生产过程实体 | `OntologyClass` (BFO:process) | ContaminationEvent、CleaningProcess、RiskAssessmentProcess |
| 关系定义 | `LinkType` | hasActiveIngredient、locatedIn、mediatedByEquipment、usedInProductionOf… |
| Action 定义 | `OntologyAction` | 专用化决策、灭活验证触发、清洁再验证触发 |

### 4.2 维护工作台（复用 + 扩展）

**复用既有组件**（infilake-dw `components/ontology/`）：
- `class-panel.tsx` — 实体类型 CRUD、审核（`is_reviewed`）、停用（`is_disabled`）、置信度展示、父类继承（`parent_class_id`）
- `link-type-panel.tsx` — 关系定义，含 `cardinality`（1:1/1:N/N:1/M:N）、`from_class → to_class`、`properties_schema`
- `action-panel.tsx` — Action 定义，含 `actor_class`、`target_class`、`preconditions`、`postconditions`、`parameters_schema`
- `graph-visualization.tsx` — ReactFlow 图谱可视化（Class 蓝节点 / Action 橙节点 / 关系带基数标签）

**领域专用扩展**：
1. **本体映射面板（新增 `ontology-mapping-panel.tsx`）**：
   - 每个 Class/LinkType 绑定 SLPRA IRI（如 `slpra-drug:CytotoxicDrug`）与 BFO 上层类（material_entity/quality/role/disposition/process/GDC）。
   - 字段级映射：`OntologyClass.properties_schema` 字段 ↔ SLPRA 数据属性（如 `pde_mg_per_day`、`oel_ug_per_m3`、`cleanabilityScore`）。
   - 提供"映射健康度"校验：未映射字段、孤立 Class、与 TTL 漂移项高亮。
2. **风险属性向导**：针对 M1 的 ~20 个风险属性（OEB、PDE、致敏等级、灭活方式…）提供受控词表录入，避免自由文本错误。
3. **TTL 导入/导出按钮**：一键从 `ontology/slpra/*.ttl` 导入校验，或将工作台变更导出为 TTL（带 diff 预览）。

### 4.3 维护流程

```
分析师登录 → 选择 Domain(clinical-shared-line-production)
  → 选择/新建 Workspace(草稿)
  → 维护 Class / LinkType / Action（带置信度与审核态）
  → 绑定 SLPRA IRI 与字段映射
  → 校验映射健康度 + 与 TTL diff
  → 提交审核 → 发布(Publish) → 导出 TTL → Git 提交
```

发布走既有 `publish_project` / `publish_workspace` 流程（Draft → Review → Published，旧版本自动归档，保留版本可追溯）。

## 5. 能力二：多源抽取与实体对齐

> **目标**：从文档（Excel/Word）、数据库实体、逻辑和 action 抽取，并完成实体对齐。

### 5.1 源类型与读取器

复用既有 `OntologySourceType`（document / data_source / ods_table）与 `source_reader.py`，**扩展文档读取器**支持 Excel 与 Word：

| 源类型 | 读取器 | 抽取内容 | 实现 |
|-------|-------|---------|------|
| **Excel（.xlsx/.xls）** | 新增 `excel_reader`（openpyxl/pandas） | 设备台账、物料清单、批记录表头→Class 字段；多 sheet→多 Class；表内行→Instance | 表头作 schema 推断，样本行作 instance |
| **Word（.docx）** | 新增 `docx_reader`（python-docx） | SOP/工艺规程正文→过程实体与 Action；表格→Class/Instance；标题层级→类层次 | 段落+表格双通道解析 |
| **数据库实体** | 既有 `read_data_source` / `read_ods_table`（DuckDB） | 表结构→Class，外键→LinkType，状态字段+时间戳→Action | 复用既有 |
| **逻辑/Action** | LLM Action 抽取 | SOP 中"若…则…必须…"句式→preconditions/postconditions；状态机→Action | 复用 `ACTION_EXTRACTION_SYSTEM` prompt |

### 5.2 抽取流水线（复用既有 5 阶段）

沿用 `extraction_service.run_extraction_background()` 的五阶段，**注入 SLPRA 领域 prompt**（domain preset），使抽取结果直接落到 SLPRA 类体系：

```
Phase 1 Classes      → 抽取实体类型，匹配 SLPRA drug/equipment/facility 分组
Phase 2 LinkTypes    → 抽取关系，匹配 hasActiveIngredient/locatedIn/usedInProductionOf…
Phase 3 Actions      → 抽取逻辑与动作（专用化决策、灭活、清洁再验证）
Phase 4 Instances    → 抽取设备/药品/区域实例（设备台账行 → CT64201 等）
Phase 5 Alignment    → 跨源实体对齐
```

**置信度**沿用加权公式：`0.4·LLM + 0.3·跨源一致 + 0.3·Schema 校验`，低于阈值（默认 0.5）的进入审核队列。

**SLPRA 领域 Preset**（新增 `OntologyDomainPreset`）：
- `seed_classes` = SLPRA 7 模块核心类
- `seed_link_types` = SLPRA 对象属性
- `seed_actions` = R-ED/R-SC/R-CP 规则对应的动作模板
- 抽取 prompt 内置 OEB/PDE/材质/洁净级别等受控词表，降低幻觉。

### 5.3 实体对齐（复用 `alignment_service.py`）

跨源实体解析（同一设备在 Excel 台账、MES、ERP 中的不同标识）：
- LLM 驱动分组 → `alignment_group` + `is_canonical` 标记权威实例。
- 复用 `alignment-review.tsx`：分析师 confirm / reject / manual_merge / split。
- **领域增强对齐键**：设备以 `equipmentID`（如 CT64201）为强主键；药品以 `API + 剂型 + 规格`（呼应 DrOn 粒度层次）为对齐键，避免分子层与产品层混淆。

### 5.4 抽取示例（设备台账 Excel）

```
输入 Excel 行：CT64201 | 离心机 | 800 | 304不锈钢衬HALAR | 一般区南平台下 | 共用
  → Instance: slpra-equip:CT64201 a Centrifuge
       constructedOf SS304WithHALARCoating ; locatedIn GeneralArea_SouthPlatform_Lower ;
       isInCleanArea false ; hasEquipmentRole SharedEquipmentRole
  → 写入既有 slpra-integration.ttl 实例区已验证模式一致
```

## 6. 能力三：实时事实源对齐与 Action 推理

> **目标**：结合本体 KG 和 Action，实时对齐生产排期(APS)、ERP、MES、临床实验管理(CTMS)等事实源进行推理。

### 6.1 事实源连接器（新增 `services/factsource/`）

| 事实源 | 提供的事实 | 接入方式 | 物化为 A-Box |
|-------|-----------|---------|-------------|
| **生产排期 APS** | 排产计划：某设备/产线在时段内生产哪些产品 | API 轮询 / Webhook | `usedInProductionOf(eq, dp)` + 时间区间 |
| **ERP** | 物料主数据、批次、产品-API 关系 | DB CDC / OData | DrugProduct / API / 批次实例 |
| **MES** | 实际执行：批次、设备状态、清洁记录、洁净区参数 | OPC-UA / REST / SSE | CleaningRecord、设备占用、压差/换气实测 |
| **LIMS** | 检验数据：残留检测、回收率、检测限 | REST | CleaningValidationProtocol 结果 |
| **CTMS** | 临床试验用药批次、试验阶段 | REST | ClinicalTrialDrug 实例、批次归属 |

**对齐策略**：连接器拉取增量 → 经能力二的映射/对齐层归一到 SLPRA IRI → 物化写入 Triple Store → 发布"事实变更事件"触发推理。延迟目标：事实变更到推理结论 **≤ 5 秒**（近实时）。

### 6.2 推理引擎（OWL + SWRL 双表示）

**OWL DL 推理（HermiT / Pellet / openllet）**：
- 自动分类：API 有 `GenotoxicityProfile` → 产品被推理为 `CytotoxicDrug`（R-DC1）。
- 一致性检查：相容性矩阵约束、污染途径不相交公理（4 途径互斥）。

**SWRL / 规则推理（复用 `slpra-swrl-rules.xml` 规则集）**：
- **规则组1 药品风险自动分类**（R-DC1~R-DC4）：基因毒/OEB≥4/致敏>3/β-内酰胺 → 分类。
- **规则组2 设备专用化决策**（R-ED1~R-ED6）：青霉素无条件专用、细胞毒+不可灭活专用、OEB5 专用、激素独立 HVAC。
- **规则组3 污染途径风险评分**（R-CP1~R-CP4）：残留+低 PDE+差可清洁性 → 高风险。
- **规则组4 CFDI 情景识别**（R-SC1~R-SC8）：自动识别 8 种共线情景。
- **冲突解决元规则**：`requiresDedication` 声明为函数性属性，true 优先（安全优先）。

### 6.3 MACO/PDE 混合计算（复用 SLPRA 第 11 节架构）

SWRL 算术能力有限 → **混合架构**：OWL 存结构与参数 → SWRL 选方法（R-MACO-1~3：PDE 可用优先 PDE 法）→ **外部计算服务**（Python）执行公式：

```
PDE = (PoD × BW) / (F1·F2·F3·F4·F5·MF)
MACO = min(1/1000 剂量法, NOEL 法, 10ppm 法, PDE 法)   # CFDI 取最小值
```

计算结果回写 `MACOCalculation.maco_mg`，作为清洁验证残留限度依据。

### 6.4 Action 引擎（事件驱动）

Action 不仅是定义，更是**可执行算子**。事实变更 → 推理 → 触发 Action：

| Action | 触发条件（前置） | 执行（后置） |
|-------|----------------|------------|
| `RaiseDedicationRequirement` | R-ED 系列推出 `requiresDedication=true` | 生成专用化要求工单 + 告警看板 |
| `TriggerInactivationValidation` | 细胞毒+可灭活共线 | 创建灭活验证任务（指派 QA） |
| `TriggerCleaningRevalidation` | 新产品使残留限度更严 / 清洁方法变更 / 成最差条件产品 | 创建清洁再验证任务 |
| `FlagShareLineConflict` | 排期使不相容产品同设备同时段 | 阻断排期 + 推送 APS 回写建议 |
| `GenerateRiskReport` | 评估完成 | 输出 `RiskAssessmentReport`（情景/专用化/风险级别/MACO） |

Action 执行通过既有任务/事件机制（`task_manager.py` 模式）+ 对外 Webhook 回写事实源（如向 APS 回写排期冲突）。

### 6.5 实时推理看板（新增 `realtime-inference-panel.tsx`）

- **共线相容性热力图**：设备 × 产品矩阵，实时着色（绿/黄/红 = compatible/conditional/incompatible）。
- **当前排期风险**：从 APS 拉取未来 N 天排产，对每个"设备-前品-后品"组合实时跑推理，列出高风险换产。
- **推理链溯源**：点击任一结论展示触发的 SWRL 规则链（呼应 SLPRA 第 13.3 节推理链示例），满足合规可解释性。

### 6.6 端到端推理示例（细胞毒药 DrugX 在共用离心机 CT64201）

```
事实源输入（实时）:
  APS: CT64201 时段 T 排产 DrugX(临床) 与 CommercialDrugY(商业)
  ERP: DrugX → API_CompoundX (PDE=0.002, OEB5, 可化学灭活)
  MES: CT64201 = 304SS+HALAR, 一般区, 可清洁性=3

推理链:
  R-DC1/R-DC2 → CytotoxicDrug + HighActivityDrug
  R-SC1 → ClinicalWithCommercialScenario
  R-SC2 → CytotoxicSharedLineScenario
  R-ED4 → requiresDedication=false + 灭活验证
  R-ED5 (OEB5) → requiresDedication=true  ⚠冲突
  冲突解决 → requiresDedication=true（安全优先）
  R-CP1 (PDE<0.01 + 可清洁性<3) → 残留污染=高风险
  MACO = min(...) = (0.002×MBS)/TDD_next  （极低限度）

Action 触发:
  RaiseDedicationRequirement（生成专用化工单）
  FlagShareLineConflict（阻断该排期 + APS 回写建议）
  GenerateRiskReport（高风险报告，强制 QA 复核）
```

## 8. 推理引擎设计

### 8.1 推理分层

```
Layer 0  本体一致性（OWL DL，HermiT）        — 启动/发布时全量校验
Layer 1  自动分类（OWL，物化推理）            — 事实变更增量
Layer 2  规则推理（SWRL/Jena Rules）          — R-DC/R-ED/R-CP/R-SC 规则集
Layer 3  数值计算（Python 计算服务）          — MACO/PDE/风险评分
Layer 4  Action 编排（事件驱动）              — 工单/告警/回写
```

### 8.2 增量推理策略

- 事实变更事件 → 仅对受影响子图（涉及设备/产品/区域）重算 Layer 1-3，避免全量重推。
- 结论写入 `InferenceResult`，与上一快照 diff，仅变化项触发 Action 与看板刷新。

### 8.3 可解释性

每条结论保存触发规则 ID 链（如 `R-DC2→R-SC2→R-ED5→冲突解决`），看板与报告可展开，满足 GMP 审计的"决策可追溯"。

| 维度 | 措施 |
