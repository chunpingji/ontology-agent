## 1. 方案概述与目标

### 1.1 业务背景

临床药物生产（尤其是**共线生产 Shared-Line Production**）面临交叉污染风险评估的信息集成挑战：药品风险属性、设备 GMP 属性、污染途径、清洁验证、设施环境、8 种 CFDI 特殊情景等知识分散在 SOP、Excel 台账、ERP/MES/LIMS/CTMS 等异构系统中，难以统一推理与实时决策。

SLPRA 本体（已建成 7 个 TTL 模块、约 720 条公理）提供了形式化的领域知识，但**缺少一个让分析师维护、从异构源抽取、并与实时事实源联动推理的工程化平台**。本方案构建该平台：**后端为独立服务**（FastAPI + Owlready2 + PostgreSQL + Python 规则引擎），**前端自建工作台组件**（能力一已落地；原设想复用 infilake-dw，实现期评估后改为自建）对接本平台 API。

### 1.2 三大核心能力（对应需求）

| 编号 | 需求 | 平台能力 | 主要承载 | 当前落地 |
|------|------|---------|---------|---------|
| **能力一** | 高级分析师维护核心药物实体、生产区域和设备实体、生产过程实体和关系定义、映射和抽取 | 本体 Schema 维护工作台（Class/LinkType/Action 的 CRUD + 审核 + 映射） | 第 4 节 | ✅ 已落地（全栈编辑维护，`001-slpra-ontology-platform`，详见 §4.0 / §14） |
| **能力二** | 从文档（Excel/Word）、数据库实体、逻辑和 action 抽取，实体对齐 | 多源抽取流水线 + 跨源实体对齐 | 第 5 节 | ✅ 已接线（002+003）：流水线触发 + SSE、DB 源读取器、对齐审核 UI、语义对齐 |
| **能力三** | 结合本体 KG 和 Action，实时对齐事实源（排期/ERP/MES/CTMS）进行推理 | 事实源连接器 + 物化 A-Box + Owlready2/规则推理 + Action 引擎 | 第 6 节 | ✅ 功能闭合（002+003）：增量物化 + Action 引擎 + 看板 + 结论状态机；**外部真实 HTTP 接入待补**（连接器现读确定性配置） |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **本体即治理契约** | SLPRA TTL 模块是 Schema 的"真理来源"，平台 Class/LinkType/Action 与之双向映射 |
| **双表示法** | OWL（结构知识，Owlready2 分类/一致性）+ **Python 规则引擎**（条件推理，专用化决策、风险评分，编码 SLPRA 的 R-DC/R-ED/R-CP/R-SC 规则语义）——沿用 OntoReg 方法论；二者构成"两层可推断"，与"可断言"的区分见 §8.0 概念澄清 |
| **人在环（Human-in-the-loop）** | 抽取/对齐结果均带 confidence，分析师审核后方可发布；高风险合规结论强制复核 |
| **事实与定义分离** | 本体定义（T-Box）由分析师维护；事实实例（A-Box）由事实源实时物化，二者通过映射层连接 |
| **安全优先冲突解决** | 多规则冲突时 `requiresDedication=true` 优先（与 CFDI"法规合规优先"一致） |
| **落地状态透明** | 前端为自建组件（能力一已落地，原设想复用 infilake-dw 未采用）；每项能力显式标注落地状态，缺口单列、纳入路线图与验收 |

### 2.1 分层职责

- **表现层（自建）**：能力一工作台为**自建** React 组件（`components/ontology/*`：class / link-type / data-property / action 面板 + restriction / mapping / risk-wizard / ttl-toolbar + d3 graph-visualization），已替换原只读页并对接**本平台独立 API**。能力二/三视图（"抽取与对齐""实时推理看板""合规签批"）亦已自建落地。**现状：`components/` 下 `ontology`(11) / `extraction`(抽取 + alignment-review) / `integration`(realtime-inference-panel) / `compliance`(qa-signature-dialog) / `reasoning` / `dashboard` 均已集成（002+003）。**
- **应用层（独立后端）**：自建 FastAPI 服务——`ontology_engine`(Owlready2)、`extraction/pipeline`、`aligner`、`reasoning/engine`(+`calculators`+`rules`)、`kg_store`、`integration/base`。**不调用 infilake-dw 后端服务或共享其数据库。**
- **存储层**：**单一 PostgreSQL**（docker-compose `db`，库名 `slpra`）承载"预置元数据表 + 运营表"；**Owlready2 OWL 存储**（SQLite World）承载 OWL/SWRL 结构推理与 SPARQL，即知识图谱存储——**不再引入独立 Triple Store（Jena/GraphDB）**，与 docker-compose 三服务形态一致。

## 8. 推理引擎设计

### 8.0 概念澄清：可断言（assert）vs 可推断（infer）

> 本节澄清贯穿 §1.3「双表示法」「事实与定义分离」与 §8.1 推理分层的一对基础概念，并据此给出平台各分类类的当前落地形态。它直接决定一条结论是"靠人记得标"还是"靠逻辑保证"。

**定义**

- **可断言（assertion）**：由分析师、抽取流水线或事实源**显式写入**的事实，系统因"被声明"而知晓——它是推理的**输入/前提**（公理）。如 `:DrugX a slpra-drug:CytotoxicDrug`。
- **可推断（inference / entailment）**：推理机依据公理或规则，从已有事实**派生出未被显式写入的新事实**——它是推理的**输出/结论**（定理），且相对前提必然成立。

**本质：必要条件 vs 充要条件**

类成员能否被推断，取决于该类的定义强度：

| 建模方式 | 公理 | 语义 | 成员判定 |
|---|---|---|---|
| 原始类 primitive | 仅 `rdfs:subClassOf`（必要条件） | `C ⊑ DrugProduct`：若 X∈C 则 X 是药品（单向） | **只能断言**——满足特征也推不出 X∈C |
| 定义类 defined | `owl:equivalentClass`（充要条件） | `C ≡ DrugProduct ⊓ ∃…`：满足条件 ⟺ X∈C（双向） | **可推断**——满足条件即自动归类 |

例（细胞毒药物的"可推断"形态）：
`CytotoxicDrug ≡ DrugProduct ⊓ ∃hasActiveIngredient.(∃hasToxicityProfile.GenotoxicityProfile)`
——只要某药品含一个带基因毒性特征的 API，推理机即**自动**得出其为 `CytotoxicDrug`，无人断言。

**对比**

| 维度 | 可断言 | 可推断 |
|---|---|---|
| 来源 | 人 / 上游显式声明 | 推理机 / 规则派生 |
| 逻辑地位 | 前提（输入） | 蕴含（输出） |
| 是否新知识 | 仅记录已说 | 显式化隐含事实 |
| 可追溯 | 靠 provenance（"谁标的"） | 有证明链（哪些公理/规则触发） |
| 维护 | 上游变更须**手动重标**，易过期 | 输入变更**自动重算**（truth maintenance） |
| 触发前提 | 无 | 类有充要条件，或有一条匹配规则 |

**本平台的"两层可推断"**

平台「双表示法」（§1.3）对应两条产生推断的路径：

1. **OWL DL 层**（Owlready2 + 可选 HermiT/Pellet，§8.1 Layer 0–1）：可推断**仅当**类为 `owl:equivalentClass` 定义类。**现状**：SLPRA 领域分类类均为原始类，故 DL 层当前承担一致性/互斥校验与层次物化，**不**自动归入风险分类。
2. **Python 规则层**（`reasoning/rules`，§8.1 Layer 2）：前向产生式规则触发后程序化 `add_class`，是平台当前"可推断"的主力；规则 ID 与 SLPRA R-DC/R-ED/R-CP/R-SC 一一对应，保留法规溯源。

**风险分类类的当前形态（务实基线）**

| 风险分类类 | 可推断路径 | 形态 |
|---|---|---|
| CytotoxicDrug 细胞毒 | R-DC1 ← API 有 `GenotoxicityProfile` | ✅ 可推断（规则层） |
| HighActivityDrug 高活性 | R-DC2 ← OEB4/OEB5 | ✅ 可推断（规则层） |
| HighSensitizingDrug 高致敏 | R-DC3 ← `sensitizationLevel > 3` | ✅ 可推断（规则层） |
| BetaLactamDrug β-内酰胺 | R-DC4 ← β-内酰胺环 | ✅ 可推断（规则层） |
| HormonalDrug 激素 | 无规则、无充要条件 | 🟡 仅可断言 |
| PenicillinDrug 青霉素 | 无规则、无充要条件（R-DC4 只产 BetaLactamDrug，且二者在 TBox 中为兄弟类，无 subClassOf） | 🟡 仅可断言 |
| 肿瘤药物 antineoplastic | 本体中无对应类 | ❌ 不可表达（仅 `CytotoxicDrug skos:closeMatch chebi:35610` 一条策展注解） |

> 开放世界假设（OWA）下，"未被断言、也未被推出"**不等于**"为假"——上述未命中项的正确状态是 **UNKNOWN**，不可坍缩为"否"。

**实务后果（为何区分要紧）**

- **GMP 可追溯**：可推断结论自带证明链（如 `R-DC1→CytotoxicDrug，据 CFDI §3.2`），满足"决策可追溯"（§8.3、§10）；纯断言仅能追溯到"谁、何时标注"。
- **数据自洽**：API 毒性数据补录后，可推断路径下次重算即自动点亮分类（§8.2 增量重算）；仅可断言的类需人工记得回标，否则漏判且无人察觉。
- **升级路径**：把 HormonalDrug/PenicillinDrug 等从"仅可断言"升级为"可推断"，即为其补 `owl:equivalentClass` 定义公理或新增 R-DC 规则；肿瘤药物则需先在本体新增 `AntineoplasticDrug` 类（可经 ATC=L01 / ChEBI:35610 物化）再加规则。

### 8.1 推理分层（基于实际栈）

```
Layer 0  本体一致性（Owlready2，可选 sync_reasoner/HermiT）  — 启动/发布时校验
Layer 1  自动分类（OWL，物化）                               — 事实变更增量
Layer 2  规则推理（Python 规则引擎，reasoning/rules ×4 组）   — R-DC/R-ED/R-CP/R-SC
Layer 3  数值计算（reasoning/calculators）                   — MACO/PDE/风险评分
Layer 4  Action 编排（事件驱动，roadmap）                     — 工单/告警/回写
```

### 8.2 增量推理策略（目标）

- 事实变更事件 → 仅对受影响子图（设备/产品/区域）重算 Layer 1-3。
- 结论写入 `reasoning_execution`（及 roadmap `inference_result`），与上一快照 diff，仅变化项触发 Action 与看板刷新。

### 8.3 可解释性

每条结论保存触发规则 ID 链（如 `R-DC2→R-SC2→R-ED5→冲突解决`），看板与报告可展开，满足 GMP"决策可追溯"。`run_assessment` 已返回 `rules_fired`（rule_id/group/inputs/conclusion/regulation_ref）。✅
