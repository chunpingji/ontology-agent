## 1. 方案概述与目标

### 1.1 业务背景

临床药物生产（尤其是**共线生产 Shared-Line Production**）面临交叉污染风险评估的信息集成挑战：药品风险属性、设备 GMP 属性、污染途径、清洁验证、设施环境、8 种 CFDI 特殊情景等知识分散在 SOP、Excel 台账、ERP/MES/LIMS/CTMS 等异构系统中，难以统一推理与实时决策。

SLPRA 本体（已建成 7 个 TTL 模块、约 720 条公理）提供了形式化的领域知识，但**缺少一个让分析师维护、从异构源抽取、并与实时事实源联动推理的工程化平台**。本方案构建该平台：**后端为独立服务**（FastAPI + Owlready2 + PostgreSQL + Python 规则引擎），**前端自建工作台组件**（能力一已落地；原设想复用 infilake-dw，实现期评估后改为自建）对接本平台 API。

### 1.2 三大核心能力（对应需求）

| 编号 | 需求 | 平台能力 | 主要承载 | 当前落地 |
|------|------|---------|---------|---------|
| **能力一** | 高级分析师维护核心药物实体、生产区域和设备实体、生产过程实体和关系定义、映射和抽取 | 本体 Schema 维护工作台（Class/LinkType/Action 的 CRUD + 审核 + 映射） | 第 4 节 | ✅ 已落地（全栈编辑维护，`001-slpra-ontology-platform`，详见 §4.0 / §14） |
| **能力二** | 从文档（Excel/Word）、数据库实体、逻辑和 action 抽取，实体对齐 | 多源抽取流水线 + 跨源实体对齐 | 第 5 节 | 🟡 流水线已建未接线 |
| **能力三** | 结合本体 KG 和 Action，实时对齐事实源（排期/ERP/MES/CTMS）进行推理 | 事实源连接器 + 物化 A-Box + Owlready2/规则推理 + Action 引擎 | 第 6 节 | 🟡 推理可用，连接器为 Stub |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **本体即治理契约** | SLPRA TTL 模块是 Schema 的"真理来源"，平台 Class/LinkType/Action 与之双向映射 |
| **双表示法** | OWL（结构知识，Owlready2 分类/一致性）+ **Python 规则引擎**（条件推理，专用化决策、风险评分，编码 SLPRA 的 R-DC/R-ED/R-CP/R-SC 规则语义）——沿用 OntoReg 方法论 |
| **人在环（Human-in-the-loop）** | 抽取/对齐结果均带 confidence，分析师审核后方可发布；高风险合规结论强制复核 |
| **事实与定义分离** | 本体定义（T-Box）由分析师维护；事实实例（A-Box）由事实源实时物化，二者通过映射层连接 |
| **安全优先冲突解决** | 多规则冲突时 `requiresDedication=true` 优先（与 CFDI"法规合规优先"一致） |
| **落地状态透明** | 前端为自建组件（能力一已落地，原设想复用 infilake-dw 未采用）；每项能力显式标注落地状态，缺口单列、纳入路线图与验收 |

### 2.1 分层职责

- **表现层（自建 + 待建）**：能力一工作台为**自建** React 组件（`components/ontology/*`：class / link-type / data-property / action 面板 + restriction / mapping / risk-wizard / ttl-toolbar + d3 graph-visualization），已替换原只读页并对接**本平台独立 API**。能力二/三的"抽取与对齐""共线风险评估""实时推理看板"视图仍为新增项。**现状：能力一组件已集成（`src/components/ontology/` 11 个）；能力二/三前端组件未建（§5/§6）。**
- **应用层（独立后端）**：自建 FastAPI 服务——`ontology_engine`(Owlready2)、`extraction/pipeline`、`aligner`、`reasoning/engine`(+`calculators`+`rules`)、`kg_store`、`integration/base`。**不调用 infilake-dw 后端服务或共享其数据库。**
- **存储层**：**单一 PostgreSQL**（docker-compose `db`，库名 `slpra`）承载"预置元数据表 + 运营表"；**Owlready2 OWL 存储**（SQLite World）承载 OWL/SWRL 结构推理与 SPARQL，即知识图谱存储——**不再引入独立 Triple Store（Jena/GraphDB）**，与 docker-compose 三服务形态一致。

## 3. 本体与知识图谱底座

### 3.1 SLPRA 模块 → 平台 Class 分组映射

平台以 **Domain：`clinical-shared-line-production`** 组织 7 个 SLPRA 模块作为 Class/LinkType 命名空间分组：

| SLPRA 模块 | 平台 Class 分组 | 核心 Class（示例） |
|-----------|----------------|------------------|
| M1 药品风险属性 | `drug` | DrugProduct, API, Residue, OEBClassification, InactivationDisposition |
| M2 GMP 设备 | `equipment` | Equipment, EquipmentSurface, ConstructionMaterial, CleanabilityRating |
| M3 污染途径 | `contamination` | ResidueContamination, MechanicalTransfer, AirborneTransmission, ConfusionEvent |
| M4 风险评估 | `risk` | RiskAssessmentProcess, MACOCalculation, 8×SharedLineScenario, CompatibilityAssessment |
| M5 清洁验证 | `cleaning` | CleaningProcess, AnalyticalMethod, CleaningValidationProtocol |
| M6 设施环境 | `facility` | ProductionArea, CleanArea(A/B/C/D), ProductionRoom, Airlock |
| 集成桥接 | `integration` | 跨模块属性 + 外部本体对齐（DrOn/ChEBI/ISA-88/IDMP/BFO） + 样例个体 |

> 后端 `OntologyEngine.MODULE_NAMES` 已登记上述 7 个模块 IRI，并在 `load()` 时先载入 `integration` 再载入各模块。✅

### 3.2 双存储一致性（目标流程 + 现状）

- **T-Box（定义）**：分析师在前端**自建工作台**维护 → 写入**可编辑元数据表**（PostgreSQL `ontology_class/link_type/data_property/action/restriction/class_mapping`，草稿为真源）→ **发布时**投影到 Owlready2 World 并由**导出器**外科式合并/更新 `ontology/slpra/*.ttl`（保留已有 ~720 公理与未建模三元组）。
  > **现状 ✅**：该流程已贯通——元数据表为编辑真源、写入经 `OntologyMetaStore`（乐观并发 + 双存储一致）、发布经 `ttl_merge` 外科式导出 + Git 提交（`001-slpra-ontology-platform`）。
- **A-Box（事实）**：事实源数据经映射 → 物化为个体写入 Owlready2 World → `kg_store` 同步影子表 `EntityShadow` → 触发推理。✅（个体 CRUD 已实现；事实源实时物化为 roadmap）
- **同步机制**：TTL 为权威基线；平台启动时**导入** TTL（`OntologyEngine.load()`）+ 投影补种元数据表；分析师变更经批次发布**导出**回 TTL（外科式合并）并走 Git 版本管理（`ontology/slpra/`）。导入✅ / 导出✅（`ttl_merge`，§9）。

### 3.3 外部本体对齐（复用 `slpra-integration.ttl`）

平台保留已建立的对齐公理：`DrugProduct ⊑ dron:00000005`、`API ⊑ chebi:23367`、`Equipment ⊑ isa88:EquipmentEntity`、`DrugProduct ⊑ idmp:MedicinalProduct` 等，使抽取/映射时可借助外部本体术语消歧。✅（TTL 已含；UI 映射编辑已落地，`ontology-mapping-panel`，§4.2）

- **规则组2 设备专用化决策**（`equipment_dedication`，R-ED1~R-ED6）：青霉素无条件专用、细胞毒+不可灭活专用、OEB5 专用、激素独立 HVAC。
- **规则组3 污染途径风险评分**（`contamination_risk`，R-CP1~R-CP4）：残留+低 PDE+差可清洁性 → 高风险。

> 说明：当前以 **Python 规则**实现条件推理（非运行期 SWRL/Jena 引擎），规则 ID 与 SLPRA 规则集一一对应，保留可解释性与法规溯源；后续如需可在 Owlready2 中加载 SWRL 做等价校验。

### 6.3 MACO/PDE 混合计算（`reasoning/calculators.py`）✅

OWL/规则负责选择方法，**Python 计算器**执行公式：

```
PDE  = (PoD × BW) / (F1·F2·F3·F4·F5·MF)            # calculate_pde
MACO = min(PDE 法, 1/1000 剂量法, 10ppm 通用限, NOEL 法)  # calculate_maco, CFDI 取最小值
```

规则选方法（PDE 可用优先 PDE 法）；结果回写 `MACOCalculation.maco_mg`，作为清洁验证残留限度依据。端点 `/api/reasoning/calculate/pde`、`/calculate/maco`、`/assess`、`/rules` 已可用。

### 7.1 存储总览（单一 PostgreSQL + Owlready2 OWL 存储）

平台**不复用 infilake-dw 后端表、不引入独立 Triple Store**。存储由两部分构成：
1. **PostgreSQL**（docker-compose `db`，库 `slpra`）：① T-Box 可编辑元数据表（Alembic 迁移建表，自建工作台的编辑真源）；② SLPRA 运营表（实际后端 models/）。
2. **Owlready2 OWL 存储**（SQLite World，`OWL_STORE_PATH`）：承载 SLPRA TTL、OWL 分类/一致性、SPARQL、A-Box 物化——即知识图谱存储。

### 7.2 PostgreSQL（预置元数据表 + 运营表）

**A. T-Box 可编辑元数据表 ✅**（由 Alembic 迁移建表，后端启动 `alembic upgrade head` 自动应用；首启再由 `project_from_ttl` 从 `ontology/slpra/*.ttl` 幂等投影补种）——支撑自建工作台的 T-Box CRUD，是编辑真源：

| 表 | 关键列 | 用途 |
|----|--------|------|
| `ontology_class` | id, slpra_iri, label, comment, parent_class_id, bfo_category, field_schema, confidence, is_reviewed, is_disabled, **version, status** | Class（T-Box 编辑真源） |
| `ontology_link_type` | slpra_iri, domain_iri, range_iri, inverse_iri, min/max_cardinality, is_functional/symmetric/transitive, version, status | 对象属性/关系定义 |
| `ontology_data_property` | slpra_iri, domain_iri, datatype, unit, controlled_vocab, version, status | 数据属性（含受控词表） |
| `ontology_action` | slpra_iri, actor_iri, target_iri, precondition, postcondition, params, version, status | Action 定义（仅定义，不触发推理，R10） |
| `ontology_restriction` | class_id, kind, property_iri, property_kind, filler_iri, cardinality, version, status | 约束/公理 |
| `ontology_class_mapping` | class_id ↔ mapping_type / target / source_system / health | 映射健康度 |
| `ontology_release` / `ontology_change_log` | release_no, status, ttl_commit_sha, published_at / entity_table, change_kind, before, after | 批次发布与变更留痕 |
| `app_user` / `app_role` | username, role_id / name（senior_analyst·operator·qa） | RBAC 主体与角色 |

> 这些表是 T-Box 的**编辑真源**；写操作经乐观并发（`version` CAS → 409）落库，发布时由 `OntologyEngine.project_entities` 投影 Owlready2 World、`ttl_merge` 外科式回写 TTL 并产生一次 Git 提交（§4.4）。已落地 ✅。

**B. SLPRA 运营表**（实际后端 `models/`，SQLAlchemy 2.0，✅已存在）：

| 表 | 模型 | 用途 |
|----|------|------|
| `entity_shadow` | `EntityShadow`(iri/class_iri/label_zh/en/module/properties_json) | A-Box 影子索引/检索（`kg_store`） |
| `extraction_config / extraction_job / extraction_candidate` | `ExtractionConfig/Job/Candidate` | 抽取配置/作业/候选 |
| `reasoning_execution / audit_log` | `ReasoningExecution / AuditLog` | 推理执行记录 / 审计留痕 |
| `integration_connector` | `IntegrationConnector` | 事实源连接配置 |

**C. 运营表扩展（roadmap）**：`fact_materialization_run`（物化批次/水位）、`inference_result`（结论快照+规则链，用于看板/审计）、`action_execution`（Action 触发记录）。

### 7.3 Owlready2 OWL 存储（取代独立 Triple Store）✅

- 启动装载 `ontology/slpra/*.ttl` 到 Owlready2 World（SQLite，`OWL_STORE_PATH`）。
- OWL 分类/一致性（可选 `sync_reasoner`→HermiT/Pellet）；SPARQL 查询；A-Box 个体增删改。
- 即"知识图谱存储"，**无需 Jena/GraphDB**；PostgreSQL 与 OWL 存储以 **IRI 主键**互联，`kg_store` 双向同步。

## 8. 推理引擎设计

### 8.1 推理分层（基于实际栈）

```
Layer 0  本体一致性（Owlready2，可选 sync_reasoner/HermiT）  — 启动/发布时校验
Layer 1  自动分类（OWL，物化）                               — 事实变更增量
Layer 2  规则推理（Python 规则引擎，reasoning/rules ×4 组）   — R-DC/R-ED/R-CP/R-SC
Layer 3  数值计算（reasoning/calculators）                   — MACO/PDE/风险评分
Layer 4  Action 编排（事件驱动，roadmap）                     — 工单/告警/回写
```

### 8.2 增量推理策略（目标）

- 事实变更事件 → 仅对受影响子图（设备/产品/区域）重算 Layer 1-3。
- 结论写入 `reasoning_execution`（及 roadmap `inference_result`），与上一快照 diff，仅变化项触发 Action 与看板刷新。

### 8.3 可解释性

每条结论保存触发规则 ID 链（如 `R-DC2→R-SC2→R-ED5→冲突解决`），看板与报告可展开，满足 GMP"决策可追溯"。`run_assessment` 已返回 `rules_fired`（rule_id/group/inputs/conclusion/regulation_ref）。✅
